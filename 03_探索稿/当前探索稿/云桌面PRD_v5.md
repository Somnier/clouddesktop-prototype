## 1. 产品目标与系统边界

### 1.1 产品目标

本产品面向学校机房、普通教学教室和考试教室场景，目标不是做一套重型 VDI，而是构建一套**以终端本地运行能力为基础、以服务器统一纳管能力为增强**的云桌面教学解决方案。

产品必须同时满足以下目标：

1. **允许脱机交付**：在学校网络未就绪时，单教室仍可完成底层安装、桌面制作、批量部署与教室终端清单导出。
2. **允许先建档后制桌面**：终端只要预装了云桌面管理系统，即使尚未制作任何桌面，也可以先导出教室终端清单并导入服务器建档。
3. **支持联机统一纳管**：服务器上线后，终端状态、配置、桌面资产与告警可逐步纳入统一管理。
4. **降低线下服务门槛**：一键替换、一键重置等高频恢复动作必须面向非研发人员可执行，且失败时恢复到上一个可正常工作的完整状态。
5. **保障课堂关键节点**：教师终端必须在平台和教室布局中被明确标识，避免影响课堂开始与继续授课。
6. **满足考试敏感场景**：考试桌面一期即需支持物理分区部署方案，以兼容对 VHD 或类似虚拟化承载方式敏感的考试软件。

### 1.2 系统边界

本产品由四类核心组成部分构成：

- **云桌面管理系统**：运行于终端裸机上的底层系统，负责制作桌面、批量部署、教室终端清单导出、一键替换/重置、网络参数修改、启动项设置等操作。
- **服务器管理平台**：运行于服务器上的 Web 控制中心，负责教室建档、终端纳管、桌面资产分发、告警展示、状态总览和系统运维。
- **桌面**：终端最终进入的运行环境。对用户和交付话术统一使用“桌面”一词；快照仅作为系统内部概念保留。
- **安装 U 盘**：交付现场使用的便携介质，用于安装云桌面管理系统并提供初始基础镜像。

本产品不以复杂版本治理、复杂 RBAC 或重型桌面包管理作为一期前提。一期遵循“能交付、能纳管、能恢复、能考试、能扩展”的原则，二期再补齐更重的管理能力。

## 2. 人员与账号模型

本项目最容易混淆的是“线下人员角色”与“平台权限账号”。两者必须分开描述。

### 2.1 线下操作者

| 人员 | 典型职责 | 说明 |
| :--- | :--- | :--- |
| **安装/机房服务人员** | 单教室部署、教师终端部署、导出清单、服务器导入、告警检查、网络改造、一键替换/重置、考试桌面制作 | 对应现实中的安装工程师、机房管理员或驻场服务人员。现实中通常掌握多个管理员账号，可根据工作职责切换使用。 |
| **教师** | 使用教师终端执行全部开机、全部关机，观察当前教室终端在线状态 | 教师不是平台后台的主要管理主体，主要使用教师终端完成课堂开关机动作。 |

### 2.2 平台账号

| 账号类型 | 管理范围 | 能力边界 | 说明 |
| :--- | :--- | :--- | :--- |
| **系统管理员账号** | 全部校区 | 拥有平台完整管理能力 | 负责平台初始化、校区级管理、全局可见。 |
| **校区子管理员账号** | 所属校区 | 与系统管理员账号的基础能力集基本一致，但可见范围受所属校区限制 | 主要用于多校区隔离场景，目标不是做复杂权限矩阵，而是做同能力、不同范围的隔离。 |

**关键判断**：

1. “机房管理员”是现实岗位概念，不宜直接作为平台权限模型名称。
2. 平台真正需要建模的是“管理员账号体系”，而不是线下服务人员的人事身份。
3. 一期应支持“系统管理员账号 + 校区子管理员账号 + 校区范围隔离”的最小闭环；复杂 RBAC 不纳入一期目标。

## 3. 核心概念与关键判断

| 概念 | 定义 | 关键判断 |
| :--- | :--- | :--- |
| **终端** | 安装云桌面管理系统的物理设备 | 终端可能只有底层系统而尚无任何桌面资产。 |
| **教师终端** | 被标记为教师用途的终端 | 教师终端在物理层面仍是普通终端，不要求成为独立硬件类型；但在服务器布局和终端详情中必须明显可见。 |
| **学生终端** | 被标记为学生用途的终端 | 与教师终端共享同一终端模型。 |
| **临时母机** | 在某个流程中临时被选作控制入口或数据源的终端 | 它不是固定资产类型，而是终端的临时工作角色。母机与教师终端都可以在特定阶段执行教室开关机动作。 |
| **基础镜像** | OS 基础模板 | 仅在需要制作桌面或分发桌面时才会产生同步，不是服务器建档的前置条件。 |
| **快照** | 基于基础镜像生成的增量层 | 快照是系统内部资产概念，用户界面不直接暴露。 |
| **桌面** | 用户最终进入的运行环境 | 标准桌面通常由“基础镜像 + 快照”构成；少数考试桌面可采用物理分区方式直接部署。 |
| **物理分区考试桌面** | 部署到真实磁盘分区的考试桌面 | 一期必须支持，用于兼容敏感考试软件。 |
| **教室终端清单** | 单教室导出的 Excel 桥梁文件 | 可在没有任何桌面资产的情况下导出，只要终端已预装底层系统并能被汇总。 |
| **服务器端终端登记** | 服务器侧维护的终端主档 | 保存教室、座位、MAC 地址列表、当前工作 MAC、终端用途标记和实时状态等信息。 |
| **终端业务配置** | 终端的 IP、计算机名、教室/座位、终端用途等业务配置 | 与服务器端终端登记双向同步。 |
| **终端运行状态** | 终端的在线状态与 CPU/内存负载等运行态信息 | 由终端主动上传至服务器。 |
| **告警记录** | 终端或服务器发现的异常信息 | 一期重点是展示，二期再补齐直接消警与基线更新。 |

## 4. 核心同步规则与回滚语义

### 4.1 核心同步规则

| 源对象 | 目标对象 | 方向 | 规则 |
| :--- | :--- | :--- | :--- |
| **教室终端清单** | **服务器端终端登记** | 单向导入 | 建档阶段即可发生，不要求终端已制作桌面。 |
| **终端运行状态** | **服务器端终端登记** | 单向上传 | 终端联机后持续上报在线状态与运行负载。 |
| **终端业务配置** | **服务器端终端登记** | 双向同步 | 单教室部署、服务器批量下发、网络改造等都会触发配置对齐。 |
| **本地基础镜像** | **服务器端基础镜像** | 按需双向同步 | 不再表达为单向上传；只有在桌面上传或桌面下发时才按需发生。 |
| **本地快照** | **服务器端快照** | 按需双向同步 | 与基础镜像一起构成桌面资产的同步闭环。 |
| **本地告警记录** | **服务器端告警记录** | 双向同步 | 一期保证告警展示同步；二期再增加处置结果回写后的基线更新能力。 |

### 4.2 桌面同步规则

1. 对用户而言，**“桌面”是分发与使用的最小业务单位**。
2. 对系统内部而言，桌面的底层通常由基础镜像与快照共同组成。
3. 当任意终端制作桌面并上传到服务器后，服务器即可将该桌面一并下发到其他终端。
4. 服务器不要求先拥有基础镜像资产，也不要求建档前必须先存在桌面资产。
5. 一期不引入显式“版本号”“发布状态”“版本回退选择”等重型治理模型。服务器只需要区分“上传中”“已就绪”“可下发”等实际工作状态。

### 4.3 回滚语义

本 PRD 中的“回滚”不表示传统意义上的版本回退，而表示：

**当一次变更类操作失败时，系统应恢复到终端上一个可正常工作的完整状态。**

据此，一期必须满足：

1. **批量部署失败**：目标终端恢复到部署前可用状态。
2. **一键替换失败**：新终端恢复到待接管的初始状态。
3. **一键重置失败**：终端恢复到重置前仍可用的完整状态，或退回到安全可恢复状态。
4. **考试桌面部署失败**：终端恢复到部署前原有可用桌面状态。

## 5. 业务生命周期

### 5.1 路径 A：预装终端先建档，后补桌面

这是本次澄清后必须纳入的正式路径：

1. 一批终端已预装云桌面管理系统，但尚未制作任何桌面。
2. 现场通过临时母机汇总终端信息，直接导出教室终端清单。
3. 将教室终端清单导入服务器，完成教室建档与终端登记。
4. 后续任一终端再制作桌面并上传至服务器。
5. 服务器将桌面下发到该教室或其他教室的目标终端。

### 5.2 路径 B：标准单教室交付

1. 选取一台终端作为临时母机。
2. 通过安装 U 盘制作教学桌面。
3. 批量部署到同教室学生终端。
4. 单独处理教师终端的个性化安装与设置。
5. 导出教室终端清单。
6. 服务器上线后导入建档。

### 5.3 路径 C：服务器辅助跨教室复用

1. 首间教室完成学生桌面和教师桌面制作。
2. 相关桌面上传至服务器。
3. 服务器向其他教室学生终端或教师终端按需下发对应桌面。

**判断**：跨教室“重复拎着 U 盘重装教师终端”不应成为长期主路径；在有服务器条件下，跨教室复用应尽量走服务器分发路径。

### 5.4 路径 D：日常变更与恢复

1. 任一终端可被选为临时母机进入编辑桌面。
2. 完成软件增量更新后，同教室先走本地分发，再走后台上传。
3. 服务器以“桌面已就绪”为条件，继续向其他教室下发。
4. 教室运行过程中，教师终端承担日常开关机动作。
5. 终端故障时通过一键替换或一键重置恢复。

### 5.5 路径 E：考试模式

1. 基于纯净桌面创建考试桌面。
2. 根据考试软件兼容性，选择快照式或物理分区式部署模式。
3. 将考试桌面批量下发到目标教室终端。
4. 锁定唯一启动项与还原策略。
5. 考后解除锁定，恢复教学桌面。

## 6. 功能需求

### 6.1 现场交付与首次建档

#### FR-01 预装终端可直接建档
- **目标**：允许已预装云桌面管理系统、但尚未制作任何桌面的终端先行纳入服务器建档。
- **执行主体**：安装/机房服务人员。
- **关键要求**：
  1. 终端只要能够被母机或临时控制端扫描与汇总，即可参与教室终端清单导出。
  2. 教室终端清单导出不以桌面制作完成为前提。
  3. 服务器导入该清单后即可建立教室布局与终端登记。
- **优先级**：1 期。

#### FR-02 母机制作教学桌面
- **目标**：在单教室内制作教学桌面，并保留纯净桌面作为考试桌面的派生基座。
- **执行主体**：安装/机房服务人员。
- **主流程**：
  1. 选取一台终端作为临时母机，插入安装 U 盘。
  2. 如终端尚未安装云桌面管理系统，则通过安装 U 盘完成底层安装。
  3. 在云桌面管理系统中点击“制作桌面”，读取安装 U 盘中的基础镜像。
  4. 生成初始桌面后，可直接编辑，也可先保留为纯净桌面，再克隆出教学桌面进入编辑。
  5. 完成教学软件安装、激活与个性化设置后，重启返回底层系统。
- **系统要求**：
  1. 支持 Windows 10/11、麒麟 OS、统信 UOS 等常见基础镜像。
  2. 教学桌面支持禁用部分快捷键与禁止自动更新等基础教学约束。
- **优先级**：1 期。

#### FR-03 单教室学生终端批量部署
- **目标**：将教学桌面批量下发到同教室学生终端，并完成座位与网络身份映射。
- **执行主体**：安装/机房服务人员。
- **主流程**：
  1. 在临时母机中点击“批量部署”，扫描同网段已上电终端。
  2. 在教室布局图中完成学生终端与座位号映射。
  3. 选择待部署桌面并开始下发。
  4. 系统自动分配 IP 地址、计算机名与座位号；若已知服务器地址，可一并写入。
- **系统要求**：
  1. 目标终端必须全屏预览分配到的座位号。
  2. 部署过程支持断点续传。
  3. 部署异常时回滚到部署前可用状态。
- **优先级**：1 期。

#### FR-04 教师终端个性化部署
- **目标**：完成教师终端的个性化桌面部署，并在系统中明确标记其教师用途。
- **执行主体**：安装/机房服务人员。
- **关键要求**：
  1. 教师终端沿用普通终端模型，不单独引入不同硬件体系。
  2. 教师终端因常伴随额外软件、品牌机配置与个性化设置，需要保留单独部署流程。
  3. 教师终端在终端业务配置与服务器端终端登记中都必须具备“教师机”标记。
  4. 教师终端完成部署后，教师桌面应至少具备教室开关机能力和终端在线状态查看能力。
- **优先级**：1 期。

#### FR-05 教室终端清单导出
- **目标**：导出服务器建档所需的标准桥梁文件。
- **执行主体**：安装/机房服务人员。
- **字段要求**：
  1. 教室名称、座位号、终端用途标记。
  2. IP 地址、计算机名、MAC 地址列表、当前工作 MAC。
  3. CPU、内存、硬盘型号与硬盘出厂唯一 SN 码。
- **关键要求**：
  1. 支持保存到安装 U 盘或本地。
  2. 文件无需二次编辑即可导入服务器。
  3. 没有桌面资产时也允许导出。
- **优先级**：1 期。

#### FR-06 服务器部署与账号初始化
- **目标**：完成服务器管理平台的安装、激活和账号体系初始化。
- **执行主体**：安装/机房服务人员。
- **主流程**：
  1. 在服务器执行自动部署脚本完成平台安装。
  2. 首次访问平台时创建系统管理员账号。
  3. 按项目需要创建校区子管理员账号。
  4. 导入授权许可证书并设置服务器 IP 地址或域名。
- **优先级**：1 期。

#### FR-07 教室清单批量导入与教室建档
- **目标**：将单教室导出的清单批量导入服务器，完成教室布局与终端主档建立。
- **执行主体**：安装/机房服务人员，或持有管理员账号的服务人员。
- **关键要求**：
  1. 批量创建逻辑教室并生成座位布局。
  2. 建立服务器端终端登记，保存 MAC 地址列表与当前工作 MAC。
  3. 保留终端用途标记，使教师终端在布局图中与学生终端可区分。
  4. 导入后立即开始对齐在线状态。
- **优先级**：1 期。

#### FR-08 服务器辅助跨教室标准化复用
- **目标**：减少多教室交付时教师终端与个性化桌面的重复安装成本。
- **执行主体**：安装/机房服务人员，或持有管理员账号的服务人员。
- **关键要求**：
  1. 首间教室完成桌面制作并上传后，服务器可向其他教室学生终端下发教学桌面。
  2. 首间教室完成教师桌面并上传后，服务器可向其他教师终端下发教师桌面。
  3. 该能力应作为多教室交付的正式主路径之一，而不是只被视为日常更新能力。
- **二期扩展**：在没有服务器的场景下，为进一步提高多教室复用效率，可考虑引入“离线桌面打包携带”能力。
- **优先级**：1 期。

#### FR-09 交付现场验证与局域网告警可见
- **目标**：在交付现场同时完成布局校验、硬件排雷与当前教室异常展示。
- **执行主体**：安装/机房服务人员。
- **关键要求**：
  1. 支持在服务器平台查看教室布局并核对座位顺序、在线状态和 IP 规划。
  2. 支持硬件不一致、外设缺失、硬盘掉速/坏道等异常的展示与导出。
  3. 支持广播风暴测试与终端带宽测速。
  4. 在母机或临时母机的局域网控制界面上，应能看到当前教室的异常提示，避免现场必须先回到服务器后台才知道问题。
- **优先级**：1 期。

### 6.2 资产视图与告警展示

#### FR-10 平台总览首页
- **目标**：提供校区级运行概览。
- **执行主体**：持有管理员账号的服务人员。
- **展示内容**：
  1. 当前校区教室总数与注册终端总量。
  2. 终端实时在线/离线比例。
  3. 服务器 CPU、内存、存储空间使用率。
  4. 当前异常告警数量与高频故障终端。
  5. 桌面上传或桌面下发的进度情况。
- **优先级**：1 期。

#### FR-11 教室布局视图与终端详情
- **目标**：让服务器平台能够以业务友好的方式展示每个终端，尤其是教师终端与多网卡信息。
- **执行主体**：持有管理员账号的服务人员。
- **关键要求**：
  1. 教师终端在布局图中必须有明确视觉标识，不应与学生终端完全等价显示。
  2. 终端详情页中，当前工作 MAC 应高亮展示；其余 MAC 地址以列表或折叠方式展示。
  3. 终端详情中应同时展示终端用途标记、在线状态、IP、计算机名和硬件信息。
- **优先级**：1 期。

#### FR-12 告警展示
- **目标**：将服务器和终端侧异常统一可见化。
- **执行主体**：安装/机房服务人员，持有管理员账号的服务人员。
- **关键要求**：
  1. 一期重点是“看见告警”。
  2. 服务器平台应展示当前校区异常概况与教室级异常明细。
  3. 母机或临时母机在现场操作时，应能看到当前教室的异常提示。
  4. 告警至少覆盖：硬件不一致、外设未连接、硬盘掉速、坏道、终端离线、服务器资源异常等。
- **优先级**：1 期。

#### FR-13 告警处置
- **目标**：为合理硬件替换或已确认异常提供消警能力。
- **执行主体**：持有管理员账号的服务人员。
- **说明**：
  1. 二期提供“接受”“忽略”等处置动作。
  2. “接受”表示当前状态被认定为合理，并可更新基线。
  3. “忽略”表示保持原始基线，但不再重复提示该条告警。
- **优先级**：2 期。

### 6.3 日常教学运行与桌面分发

#### FR-14 教室开关机控制
- **目标**：支持教师终端和临时母机对学生终端执行全部开机、全部关机。
- **执行主体**：教师，安装/机房服务人员。
- **关键要求**：
  1. 日常上课场景下，由教师终端承担主要入口。
  2. 交付、验收或排障场景下，临时母机也可执行同样的开关机控制。
  3. 当前需求仅覆盖开机/关机，不强行扩展为更重的“群控套件”。
- **优先级**：1 期。

#### FR-15 临时母机本地编辑与同教室分发
- **目标**：支持在单教室内完成桌面增量更新，并优先通过本地路径完成快速分发。
- **执行主体**：安装/机房服务人员。
- **主流程**：
  1. 任意终端可被选为临时母机，进入编辑桌面。
  2. 完成软件安装、激活、验证后，返回云桌面管理系统。
  3. 系统合并本次桌面修改结果。
  4. 先向同教室其他终端执行本地批量部署。
  5. 再在后台将桌面上传到服务器。
- **关键要求**：后台上传进度必须对用户可见。
- **优先级**：1 期。

#### FR-16 服务器按需下发桌面
- **目标**：服务器在桌面上传完成后，将桌面下发到目标终端，包括学生终端与教师终端。
- **执行主体**：持有管理员账号的服务人员。
- **关键要求**：
  1. 对用户而言，下发对象是“桌面”，而不是单独的基础镜像或快照。
  2. 对系统内部而言，下发过程需要一并处理基础镜像与快照的按需同步。
  3. 服务器只需区分“上传中”“已就绪”“可下发”等状态，不引入显式版本发布流程。
  4. 上传未完成时，服务器必须展示进度且不允许误下发。
- **优先级**：1 期。

### 6.4 故障恢复

#### FR-17 一键替换
- **目标**：在原工位硬件损坏时，让新终端快速接管原有身份和桌面。
- **执行主体**：安装/机房服务人员。
- **主流程**：
  1. 将新终端接入原座位的网线与电源并开机。
  2. 在底层系统中点击“一键替换”，输入服务器地址。
  3. 选择需替换的教室与座位。
  4. 系统下发原座位的业务配置和桌面资产。
  5. 新终端完成上报后，服务器更新其 MAC 地址列表、当前工作 MAC 与硬件信息。
- **关键要求**：
  1. 新终端可替换学生终端，也可替换教师终端。
  2. 失败时必须恢复到待接管初始状态。
- **优先级**：1 期。

#### FR-18 一键重置
- **目标**：在终端遭遇蓝屏、中毒、异常卡顿等软件故障时，恢复到上一个可正常工作的完整状态。
- **执行主体**：安装/机房服务人员，教师。
- **关键要求**：
  1. 支持从服务器或本地可用缓存恢复桌面与配置。
  2. 整个恢复过程可视化展示进度。
  3. 失败时恢复到重置前仍可工作的完整状态，或退回到安全可恢复状态。
- **优先级**：1 期。

### 6.5 校园网改造

#### FR-19 终端 IP 批量变更
- **目标**：在学校网段变更时，按座位顺序批量重写终端 IP 配置。
- **执行主体**：安装/机房服务人员，持有管理员账号的服务人员。
- **入口**：
  1. 单教室入口：临时母机中的“批量修改 IP 地址”。
  2. 服务器入口：服务器平台中的“批量修改 IP 地址”。
- **关键要求**：修改后终端业务配置与服务器端终端登记保持一致。
- **优先级**：1 期。

#### FR-20 服务器地址变更
- **目标**：在服务器 IP 地址或域名变更后，尽量避免终端失联。
- **执行主体**：安装/机房服务人员，持有管理员账号的服务人员。
- **关键要求**：
  1. 必须遵循“先通知终端，后修改服务器本体”的顺序。
  2. 支持服务器端向在线终端推送新地址。
  3. 支持单教室临时母机作为漏通知终端的补救入口。
  4. 若启用 HTTPS，地址变更后支持导入新 SSL 证书。
- **优先级**：1 期。

### 6.6 考试环境

#### FR-21 考试桌面制作
- **目标**：基于纯净桌面制作独立考试桌面。
- **执行主体**：安装/机房服务人员。
- **关键要求**：
  1. 支持从纯净桌面克隆出考试桌面。
  2. 支持进入编辑桌面完成考试软件安装和验证。
  3. 考试桌面与日常教学桌面必须独立管理。
- **优先级**：1 期。

#### FR-22 物理分区式考试桌面
- **目标**：兼容不接受 VHD 或类似虚拟化承载方式的敏感考试软件。
- **执行主体**：安装/机房服务人员。
- **关键要求**：
  1. 一期必须支持将考试桌面部署到真实物理分区。
  2. 在业务操作层面，仍应尽量沿用“制作桌面、批量部署、锁定启动项”的统一心智模型。
  3. BIOS 引导策略需要能够把该物理分区桌面设置为加载目标。
  4. 当考试桌面采用物理分区方式时，系统仍需支持后续批量部署与恢复流程。
- **优先级**：1 期。

#### FR-23 考试环境锁定
- **目标**：在考试前将终端限制为仅可进入考试桌面，并设定正确的还原策略。
- **执行主体**：安装/机房服务人员。
- **关键要求**：
  1. 将考试桌面设置为唯一启动项，并隐藏其他桌面入口。
  2. 根据考试要求设置还原模式，通常为“系统盘还原、数据盘保留”。
  3. 对物理分区式考试桌面，启动项必须直接指向物理分区桌面。
  4. 考后可手动解除锁定并恢复教学桌面。
- **优先级**：1 期。

### 6.7 账号与审计

#### FR-24 校区子管理员隔离
- **目标**：在多校区场景下实现同能力、不同可见范围的账号隔离。
- **执行主体**：系统管理员账号。
- **关键要求**：
  1. 系统管理员账号可见全部校区。
  2. 校区子管理员账号仅可见所属校区的教室和终端。
  3. 两者的基础能力集尽量保持一致，降低培训与使用成本。
  4. 一期只做校区范围隔离，不做复杂 RBAC。
- **优先级**：1 期。

#### FR-25 操作日志
- **目标**：支持问题追踪与责任溯源。
- **执行主体**：系统自动记录。
- **关键要求**：
  1. 服务器平台记录管理员账号的关键操作行为。
  2. 终端本地记录关键操作行为，包括桌面上传、替换、重置、批量部署、教室开关机等动作。
  3. 日志字段至少包含：操作人、账号、时间、操作对象、动作内容、执行结果。
- **优先级**：2 期。

#### FR-26 离线桌面打包携带
- **目标**：在没有服务器的多教室交付场景下，提高教师终端和个性化桌面的跨教室复用效率。
- **执行主体**：安装/机房服务人员。
- **说明**：该能力会引入新的桌面携带与导入复杂度，不纳入一期，作为二期效率增强方向保留。
- **优先级**：2 期。

| 阶段 | 建议纳入范围 |
| :--- | :--- |
| **1 期** | FR-01 至 FR-12、FR-14 至 FR-24 |

## 8. 本版相对早期 PRD 的关键修正

1. 不再把早期 PRD 的章节骨架视为约束，而是按业务生命周期重构文档结构。
2. 将“线下人员角色”与“平台账号模型”彻底分开，避免把机房管理员误写成系统权限角色。
3. 明确“服务器建档不以桌面资产存在为前提”。
4. 将基础镜像与快照的服务器/终端关系修正为“随桌面按需双向同步”，不再表达为固定单向上传。
5. 将教师终端明确为“普通终端 + 教师用途标记 + 教师桌面能力”，同时要求在服务器布局中明显可见。
6. 将告警能力拆为“一期展示、二期处置”。
7. 将物理分区式考试桌面从兼容性备注提升为一期正式需求。
8. 删除“一期显式版本治理”的假设，回到更符合业务现状的“上传状态 + 可用状态 + 回滚到上一个完整可工作状态”的表达。

---

> **结论**：本版 PRD 已可作为研发、测试、实施与后续原型设计的共同基线。若继续深化，下一步最值得展开的是三项：
> 1. 服务器端教室布局与终端详情页的信息架构。
> 2. 教师终端标识、MAC 展示与告警展示的具体 UI 方案。
> 3. 物理分区式考试桌面的交互路径与原型约束。
