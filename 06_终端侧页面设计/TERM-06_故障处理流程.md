## 1. 这页真正要解决什么问题

TERM-06 需要把一键替换和一键重置彻底分开，不再混成一个模糊的“恢复一下”。

这页要解决两类完全不同的问题：

1. 一键替换：一台新的普通终端接入原工位后，接管原终端的业务身份和桌面资产。
2. 一键重置：当前故障终端从服务器重新拉取自己已登记的桌面与配置，并整机重刷。

这里有两个必须写清的前提：

1. 一键重置当前不再依赖本地缓存。服务器是权威恢复源。
2. 终端本地当然会保存运行所需的镜像、桌面和快照数据，但这些属于本地运行数据，不应轻率被叫作“缓存恢复源”。

## 2. 这页与其他页面的关系

1. TERM-06 的主入口在 [TERM-01_管理系统首页.md](./TERM-01_管理系统首页.md) 的“故障处理”。
2. TERM-06 主要服务单终端本地流程，不以母机工作页为前提。
3. TERM-06 不是母机批量任务执行中的并行分支；受控终端在执行中不能自行跳到这里。
4. 如果后续要从 TERM-03 提供补充入口，也应跳到本机故障处理视角，而不是把 TERM-06 变成教室级批量页；这页讨论的是脱离整批任务后的单机排障，不再回到 TERM-04、TERM-05、TERM-07 的批量终端心智。
5. 一键替换和一键重置都依赖服务器地址和服务器侧登记信息；如果这些前提不成立，应在当前流程里直接展开“网络与服务器”字段组修正，而不是把用户打断到独立页面。

补充边界：

1. 当前终端未受控时，可直接进入 TERM-06。
2. 当前终端已受控且任务仍在执行时，不允许进入 TERM-06。
3. 只有当当前任务已中断、已取消，或当前终端已经明确脱离整批任务后，TERM-06 才重新成立。

## 3. 当前页依赖的数据层

### 3.1 本机当前信息

1. 当前工作 MAC。
2. 当前 IP 地址。
3. 本机是否已登记为某个既有终端。
4. 当前服务器地址状态。
5. 当前生效网络配置，以及终端业务配置中保存的网络配置。

### 3.2 一键替换相关数据

1. 服务器是否返回高置信替换候选。
2. 原教室和原座位。
3. 原座位的业务配置和桌面资产摘要。
4. 手动选择教室和座位的可选项。
5. 原终端登记关系如何转交给当前新终端。

### 3.3 一键重置相关数据

1. 终端是否已注册到服务器（单一前置条件，不再分别检查"已注册桌面"和"已注册配置"）。
2. 当前终端是否具备访问服务器并重新拉取资源的前提（服务器可达）。
3. 重置进度和结果。

### 3.4 风险提示数据

1. 是否正在绕过替换流程。
2. 绕过后会覆盖哪类历史登记关系。
3. 失败原因和原始错误详情。

## 4. 页面字段定义

| 字段 | 含义 | 来源 | 何时可显示 | 说明 |
| :--- | :--- | :--- | :--- | :--- |
| 处理模式 | 当前选择的是一键替换还是一键重置 | 页面状态 | 进入 TERM-06 后 | 两条链路必须分开表达 |
| 服务器前提状态 | 当前终端是否已具备访问服务器的前提 | 本机当前信息 | 进入后 | 一键替换和一键重置都依赖它 |
| 网络与服务器修正区 | 当前流程内可直接修改的 IP、掩码、网关、DNS、服务器地址字段组 | 本机当前信息 | 前提不满足或用户主动展开时 | 这是 TERM-01“网络与服务器”的内联复用；既要回显当前生效值，也要回显业务配置值，不应把用户打断到别处 |
| 替换候选 | 服务器返回的高置信原座位候选 | 一键替换相关数据 | 有候选后 | 正常路径应优先自动建议 |
| 手动替换目标 | 用户手动选择的教室和座位 | 一键替换相关数据 | 进入手动流程后 | 作为自动建议的补充入口 |
| 当前终端注册状态 | 当前终端是否已注册到服务器 | 一键重置相关数据 | 一键重置前 | 单一前置条件，不分别检查桌面和配置 |
| 服务器重置前提 | 服务器是否可达，且终端已注册 | 一键重置相关数据 | 确认前 | 一键重置需要服务器可达 + 终端已注册 |
| 绕过替换风险 | 用户不走替换而继续普通使用的后果 | 风险提示数据 | 用户选择跳过替换时 | 需要明确提示可能覆盖原登记 |

## 5. 流程骨架

### 5.1 先检查服务器前提，但要在当前流程里改

1. 一键替换先检查服务器地址是否已配置。
2. 一键重置也先检查服务器地址和服务器可达性。
3. 如果服务器前提不成立，应在当前流程内直接展开 IP、掩码、网关、DNS 和服务器地址字段，让用户保存并重新检测；如果当前生效值与业务配置值不一致，也要在这里一起表达，而不是跳到另一个独立配置入口。
4. 如果服务器不存在或暂时不可达，检测应快速返回失败结果，不长时间阻塞页面，但不把具体秒数写成硬规则。

### 5.2 一键替换以自动建议为主，手动选择为补充

1. 正常情况下，如果服务器能返回高置信候选，应直接提示“是否替换原座位”。
2. 手动选择教室和座位是补充入口，不是默认主链路。
3. 如果用户放弃替换继续普通使用，系统必须提示这会覆盖或打断原有登记连续性。
4. 替换成功后，服务器应保留原终端的历史 MAC 关系，但把当前新终端 MAC 记为新的当前工作 MAC；旧终端不再继续保留原座位的当前身份。

### 5.3 一键重置只描述当前实际动作

1. 一键重置不是抽象的“恢复到最近可工作状态”。
2. 一键重置实际执行的是“把服务器侧为当前终端登记的桌面与配置重新拉取并重刷到当前终端”。
3. 页面文案必须围绕这个实际动作表达，不再把本地缓存写成可信恢复源。
4. 如果服务器登记存在，但对应资源当前不齐，不应强行进入重置进度页，而应先把它表达成“当前不能开始重置”。

### 5.4 本地数据和缓存不是一个概念

1. 当前终端本地当然会保存镜像、桌面和快照等运行所需数据，否则本机无法启动和进入桌面。
2. 但当前产品口径不把这些本地数据再额外包装成“缓存恢复源”。
3. 当前一键重置只认服务器登记和服务器下发结果。

## 8. 页面结论

1. 一键替换和一键重置的对象、前提和文案都必须彻底分开。
2. 一键替换的主路径是新终端本地自动提示，不是先到母机再手动挑。
3. 一键重置当前要明确成“依赖服务器重刷当前终端已登记内容”，不再把本地数据叫成可信缓存恢复源。
4. 服务器地址、IP、网关、DNS 等前提修正必须内联在当前故障处理流程里，不应把用户打断到独立配置孤岛。
5. TERM-01 的“网络与服务器”是单终端常驻入口，TERM-06 是在故障处理意图下对同一字段组的流程内复用。
