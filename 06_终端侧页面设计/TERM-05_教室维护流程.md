## 1. 这页真正要解决什么问题

TERM-05 承接的是“当前教室已经交付后，怎么继续维护这批终端”，但它内部其实只有两类维护动作：

1. 桌面更新类维护，例如加装软件、更新授权、调整桌面内工具。
2. 轻量配置类维护，例如调整终端网络配置、更新终端记录的服务器地址。

这两类动作可以放在同一页承接，但不能被写成同一种东西。

TERM-05 必须同时守住下面几个边界：

1. 当前教室受管集合，是这间教室当前被母机管理的终端总集合。
2. 这轮勾选终端，是这次维护真正要处理的终端集合。
3. 保持现状终端，是留在教室里但这轮不动的终端。
4. 如果维护涉及桌面内容，这轮待下发桌面集合可以是当前母机已有的一个或多个本机桌面。
5. 当前母机需要改桌面内容时，应先处理本机桌面，再进入这轮桌面更新维护。
6. 真正下发维护结果时，处理的对象仍然是一批终端，而不是一批桌面。

因此 TERM-05 不是在创建什么“维护计划”。原型真正要表达的是：用户先选这轮要维护哪些终端，再决定这轮维护的内容，系统给出逐台预览和校验，确认后再开始执行。

## 2. 这页与其他页面的关系

1. TERM-05 从 [TERM-03_母机工作页.md](./TERM-03_母机工作页.md) 的“教室维护”进入。
2. 如果这轮维护涉及桌面内容，TERM-05 与 [TERM-04_单教室部署流程.md](./TERM-04_单教室部署流程.md) 共享“选终端 -> 选已有本机桌面 -> 逐台确认 -> 本地分发”的纯批量骨架；桌面制作动作不再并进 TERM-05 的批量选择层，但桌面选择步骤应提供单独的 [修改本机桌面] 按钮。点击后终端重启进入当前母机的本机桌面处理；在那里可新建桌面、复制已有桌面，或进入桌面编辑；完成这次桌面处理并再次重启后，系统回到 TERM-05 当前步骤。
3. 如果这轮维护只是轻量配置，TERM-05 不进入桌面，但仍然要保留“选择终端 -> 预览写入结果 -> 校验 -> 批量执行”的批量任务骨架。
4. TERM-05 执行期间，受控终端仍停留在 [TERM-01_管理系统首页.md](./TERM-01_管理系统首页.md) 的受控任务屏，只表达本机进度，不回看整批视角。
5. 当前教室本地执行完成，与服务器侧是否同步完成，是两条并行线；页面口径以 [云桌面终端与服务器按需同步说明.md](../05_云桌面事实/云桌面终端与服务器按需同步说明.md) 为准。
6. 如果当前批量维护里只剩个别终端需要单独替换、重置或改网络，应转交 [TERM-06_故障处理流程.md](./TERM-06_故障处理流程.md) 的单机排障视角继续处理。
7. 这轮不参与的终端仍属于当前教室受管集合；后续如需单机处理，再进入对应本机入口或故障处理流程，TERM-05 本身不展开后续补路说明。

## 3. 当前页依赖的数据层

### 3.1 当前教室上下文

1. 当前教室名称。
2. 当前母机身份。
3. 当前受管终端集合。
4. 当前在线终端数。
5. 当前教师终端数量。
6. 最近刷新时间。

### 3.2 当前维护模式与范围

1. 当前选择的是桌面更新类还是轻量配置类。
2. 当前教室受管集合。
3. 这轮勾选终端。
4. 保持现状终端。
5. 当前是否仍有终端保持现状。
6. 当前终端排序。
7. 当前风险确认状态。

### 3.3 当前母机本机桌面选择状态

1. 当前母机已有桌面列表，至少展示桌面名称、时间和备注。
2. 当前勾选的待下发桌面集合。
3. 桌面更新确认列表。
4. 本地分发状态。
5. 如果当前教室已配置服务器，后台上传状态。

### 3.4 轻量配置链路状态

1. 当前生效网络配置。
2. 终端业务配置中保存的网络配置。
3. 当前准备写入的新网络配置。
4. 当前服务器地址、准备写入的新服务器地址。
5. 逐台预览结果。
6. 执行前校验结果。
7. 推送状态。

### 3.5 任务结果与失败信息

1. 成功、进行中、失败、未开始终端数量。
2. 标准失败原因码。
3. 原始错误详情入口。

## 4. 页面字段定义

| 字段 | 含义 | 来源 | 何时可显示 | 说明 |
| :--- | :--- | :--- | :--- | :--- |
| 维护类别 | 当前选择的维护动作大类 | 当前维护模式与范围 | 进入 TERM-05 后 | 至少区分“桌面更新”和“轻量配置” |
| 这轮勾选终端 | 这次维护真正要处理的终端集合 | 当前维护模式与范围 | 勾选后 | 与当前教室受管集合分开表达 |
| 保持现状终端 | 当前教室里这轮不动的终端集合 | 当前维护模式与范围 | 勾选后 | 不参与这轮执行，但仍在教室上下文中 |
| 当前顺序 | 这轮勾选终端在本页里的处理顺序 | 当前维护模式与范围 | 勾选后 | 网络配置等批量写入仍可能依赖这个顺序，且应支持手动拖动调整 |
| 维护桌面集合 | 这轮桌面更新要下发的一个或多个桌面 | 当前母机本机桌面选择状态 | 选中后 | 允许多选 |
| 当前母机桌面列表 | 当前母机可直接用于这轮更新的本机桌面集合 | 当前母机本机桌面选择状态 | 进入桌面更新后 | 只负责选择已有桌面 |
| 桌面更新确认列表 | 逐台确认最终要写入什么 | 当前母机本机桌面选择状态 | 选好桌面后 | 分发前最后一层确认 |
| 网络配置预览 | 当前值、配置值和这次写入值的对照 | 轻量配置链路状态 | 执行前 | 不能只显示准备写入的新值 |
| 服务器地址推送预览 | 当前服务器地址与新地址的对照 | 轻量配置链路状态 | 执行前 | 需要明确作用终端范围；这里只改终端记录的地址，不下发客户端证书 |
| 本地分发状态 | 当前教室本地执行结果 | 当前母机本机桌面选择状态或轻量配置链路状态 | 发起后 | 决定当前教室是否已经完成 |
| 服务器上传状态 | 当前教室结果是否继续同步到服务器 | 当前母机本机桌面选择状态 | 当前已配置服务器且服务器可用时 | 不反向改变本地已完成结果 |
| 失败原因 | 当前失败终端对应的标准原因 | 任务结果与失败信息 | 失败后 | 不能只写“写入失败” |

## 5. 流程骨架

### 5.0 教室维护前提要按维护类型判断

1. 桌面更新类维护依赖当前母机已经准备好的本机桌面；如果当前还没有合适桌面，或某个桌面内容还要继续准备，可在桌面选择步骤点击 [修改本机桌面]；在那里可新建桌面、复制已有桌面，或进入桌面编辑；完成这次桌面处理并再次重启后，系统回到 TERM-05 当前步骤。
2. 批量改终端 IP、批量改服务器地址这两类轻量配置，不要求当前母机已有桌面；只要当前母机还能和这间教室的受管终端通信，就可以继续。
3. 因此 TERM-05 的入口判断不是“当前母机有没有桌面”这一把锁，而是看当前维护动作真正依赖什么。

### 5.1 先选维护类别，再选这轮要处理的终端

1. 先决定这轮维护属于桌面更新还是轻量配置。
2. 再从当前教室受管集合里勾选这轮要处理的终端。
3. 页面必须同时显示这轮勾选终端和保持现状终端，不能只给一个总数。
4. 如果教室里存在教师终端，也是在同一终端列表里勾选参与或保持现状，不单独拆一个额外流程。
5. 只要终端列表需要人工勾选，就应支持字段排序、分页和已勾选终端拖动排序；可排序字段、默认排序字段和分页策略按当前维护场景决定，不写死。
6. 不参与这轮的终端，在确认页和结果页里只需要明确写出本次不修改，并回显沿用了什么当前内容。

### 5.2 桌面更新类维护必须沿用 TERM-04 的边界

1. 先在当前母机已有桌面列表中勾选 1 个或多个要下发的目标桌面。
2. 如果当前母机还缺目标桌面，或桌面内容还要继续准备，应在当前步骤点击 [修改本机桌面]，进入当前母机的本机桌面处理；在那里可新建桌面、复制已有桌面，或进入桌面编辑；完成这次桌面处理并再次重启后，系统回到 TERM-05 当前步骤。
3. 选择完维护桌面后，不是立刻执行，而是先回到逐台确认页，核对这轮勾选终端、保持现状终端、维护桌面集合和覆盖策略。
4. 确认无误后，再向这轮勾选终端发起本地分发。
5. 如果当前教室已配置服务器且服务器可用，可在本地分发后继续后台上传，但这不阻塞当前教室本地完成。
6. 执行中必须支持主动取消；取消后已完成终端保持当前结果，未完成终端不按成功处理。
7. 只要桌面列表需要人工勾选，就应支持排序和分页；可排序字段和分页策略按当前桌面规模决定，不写死。

### 5.3 轻量配置类维护不进入桌面，但仍要逐台预览

1. 调整终端网络配置时，必须同时展示当前生效值、业务配置值和这次写入值。
2. 更新服务器地址时，必须同时展示当前地址、新地址和作用终端范围。这里改的是终端业务配置中记录的服务器地址，不向客户端下发证书。
3. 只要仍存在“范围为 0”“新地址为空”“地址格式非法”“IP 冲突”这类问题，就不能开始执行。
4. 轻量配置同样先预览这轮勾选终端，再批量执行，不应跳过确认层。
5. 执行中同样必须支持主动取消。

### 5.4 结果页必须能解释部分成功

1. 已完成终端、失败终端和未开始终端必须拆开显示。
2. 终端未响应、网络中断、服务器不可达、未返回可识别原因应分开表达。
3. 当前教室本地完成但服务器上传失败时，也要明确说明“不影响当前教室已完成结果”。
4. 如果当前存在保持现状终端，结果页继续按“本次不修改”显示，不把它们并进失败终端。

## 8. 页面结论

1. TERM-05 不是维护计划页，而是“选这轮终端 -> 选维护内容 -> 逐台确认 -> 批量执行”的现场维护页。
2. 只要维护涉及桌面内容，这轮待下发桌面集合可以多选，但来源只能是当前母机已经准备好的本机桌面。
3. 桌面更新沿用 TERM-04 的纯批量骨架：桌面选择步骤提供 [修改本机桌面] 按钮，供当前母机补改桌面；批量步骤本身只负责选已有桌面、逐台确认和本地分发。
4. 当前教室本地完成，与服务器侧是否同步完成，要拆成两条状态线表达。
5. 更新服务器地址时，终端侧只改业务配置中的服务器地址，不向客户端下发证书；若启用 HTTPS，新证书由服务器侧处理。
6. 维护失败必须能解释部分成功、已完成终端和未完成终端，不再用空泛结果文案。
