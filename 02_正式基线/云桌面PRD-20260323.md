## 1. 产品目标与系统边界

### 1.1 产品目标

本产品面向学校机房、普通教学教室和考试教室场景，目标不是构建一套重型 VDI，而是提供一套**以终端本地运行能力为基础、以服务器统一纳管能力为增强**的云桌面教学解决方案。

本产品需同时满足以下目标：

1. **支持脱机交付**：在学校网络尚未就绪时，单教室仍可完成底层安装、桌面制作、批量部署与教室终端清单导出。
2. **支持联机纳管**：服务器上线后，终端状态、终端配置、桌面资产与异常信息可逐步纳入统一管理。
3. **支持多教室复用**：多教室交付默认可重复单教室部署流程，服务器辅助下发和离线打包用于进一步提升效率。
4. **支持交付后自助运维**：交付完成后的常见操作应足够清晰，使机房管理员能够在电话指导、第三方远程工具或 VPN 接入平台等远程支持方式下快速掌握。
5. **支持教学桌面与考试桌面并存使用**：考试桌面可与教学桌面按需分别制作、隐藏、下发、设置启动项和恢复；对不接受 VHD 或类似承载方式的考试软件，在二期补充物理分区兼容路径。
6. **降低恢复门槛**：终端故障恢复能力应面向机房管理员可执行，尽量避免频繁依赖实施人员到场。

### 1.2 系统组成

本产品由四类核心组成部分构成：

- **云桌面管理系统**：运行于终端裸机上的底层系统，负责制作桌面、批量部署、教室终端清单导出、一键替换、一键重置、网络参数修改、启动项设置等操作。
- **服务器管理平台**：运行于服务器上的 Web 控制中心，负责教室创建、终端纳管、桌面下发、告警展示、状态总览、测试视图和系统运维。
- **桌面**：终端最终进入的运行环境。对用户和交付话术统一使用“桌面”一词；快照仅作为系统内部概念保留。
- **安装 U 盘**：交付现场使用的便携介质，用于安装云桌面管理系统并提供初始基础镜像。

### 1.3 设计原则

1. **术语克制**：界面与文档面向业务沟通时统一使用“桌面”等固有词汇，不引入不必要的新造词。
2. **可视化校验**：涉及教室座位映射、教师终端识别、批量部署与异常查看时，优先以布局图和终端屏幕预览辅助核对。
3. **长任务透明**：批量部署、桌面上传、服务器下发、考试环境部署等耗时动作都应展示进度和阶段状态。
4. **角色收敛**：交付后常见操作默认应可由机房管理员在远程指导下完成，技服工程师不应成为日常操作前提。
5. **测试分层**：单教室完成交付后即可执行本地硬件与基础网络测试；广播风暴测试必须明确要求至少两间已交付教室共同参与；临时母机当前教室界面和服务器平台都应具备相应测试入口。
6. **安全结束**：涉及替换、重置、批量部署等高风险动作时，系统应避免把终端留在不可用中间态。

<a id="people-and-accounts"></a>
## 2. 人员与账号模型

### 2.1 线下角色

| 角色 | 主要职责 | 说明 |
| :--- | :--- | :--- |
| **安装工程师** | 项目交付、单教室部署、教师终端部署、服务器部署、教室创建与终端登记、交付测试与验收 | 对应项目交付阶段的主要执行人。 |
| **机房管理员** | 日常开关机、加装软件、跨教室桌面下发、常见故障恢复、常规网络改造、日常平台查看 | 对应学校侧日常运维人员。 |
| **技服工程师** | 考试部署支持、少量复杂变更指导、客户关系维护 | 原则上不作为交付后日常操作前提，优先通过电话指导、第三方远程工具或 VPN 接入平台等方式支持机房管理员。 |
| **教师** | 使用教师终端执行全部开机、全部关机，观察本教室终端在线状态 | 主要参与课堂开始和结束的高频动作。 |

### 2.2 平台账号

| 账号类型 | 可见范围 | 说明 |
| :--- | :--- | :--- |
| **系统管理员账号** | 全部校区 | 用于平台初始化、全局查看和管理。 |
| **校区子管理员账号** | 所属校区 | 与系统管理员账号基础能力集基本一致，主要差异在可见范围。 |

### 2.3 使用原则

1. 线下工作按角色描述，平台权限按账号类型描述。
2. 文档正文优先使用安装工程师、机房管理员、技服工程师、教师这四类线下角色，避免反复引入新的服务主体名称。
3. 平台相关操作默认通过系统管理员账号或校区子管理员账号执行，具体由现场持有账号的人员根据职责切换；交付完成后的常见操作应优先设计为机房管理员可在电话指导、第三方远程工具或 VPN 接入平台等远程支持方式下完成。

<a id="core-concepts"></a>
## 3. 核心概念与数据流转原则

### 3.1 核心概念

| 概念 | 定义 | 说明 |
| :--- | :--- | :--- |
| **终端** | 安装云桌面管理系统的物理设备 | 教师终端和学生终端共享同一终端模型。 |
| **教师终端** | 被标记为教师用途的终端 | 在物理层面仍是普通终端，但在平台展示和终端配置中必须明确标识。 |
| **学生终端** | 被标记为学生用途的终端 | 与教师终端共享同一终端模型。 |
| **临时母机** | 在某个流程中临时被选作控制入口或数据源的终端 | 不是独立资产类型，而是终端在特定阶段承担的工作角色。 |
| **基础镜像** | OS 基础模板 | 主要来源于安装 U 盘，也可随桌面同步到服务器或从服务器下发。 |
| **快照** | 基于基础镜像生成的增量层 | 仅作为系统内部资产概念保留，不直接面向用户展示。 |
| **桌面** | 用户最终进入的运行环境 | 标准桌面通常由基础镜像与快照构成；少量考试桌面在二期支持物理分区部署。 |
| **教室终端清单** | 单教室导出的桥梁文件 | 用于服务器创建教室、登记终端与导入桌面包。 |
| **桌面包** | 跨教室复制使用的离线携带单元 | 支持导入导出，可通过 U 盘携带到其他教室继续部署。 |
| **服务器端终端登记** | 服务器侧维护的终端登记信息 | 保存教室、座位、MAC 地址列表、当前工作 MAC、终端用途、在线状态等信息。 |
| **终端业务配置** | 终端的 IP、计算机名、教室/座位、终端用途等业务配置 | 与服务器端终端登记双向同步。 |
| **终端运行状态** | 终端在线状态与 CPU/内存负载等运行态信息 | 由终端主动上传到服务器。 |
| **告警记录** | 终端或服务器自动采集、比对或检测后发现的异常信息 | 一期重点是展示，二期补充处置动作。 |

### 3.2 数据流转原则

1. **教室终端清单先于桌面资产也可以成立**：正常交付流程通常先制作桌面再导出清单，但系统也应兼容尚未制作桌面时先导出清单、再导入服务器创建教室的场景。
2. **桌面是分发与使用的最小业务单位**：对用户而言，服务器下发的是“桌面”，而不是单独的基础镜像或快照。
3. **基础镜像与快照随桌面按需同步**：当终端上传桌面或服务器下发桌面时，系统内部同时处理基础镜像和快照的同步。
4. **终端状态与终端配置分层处理**：终端运行状态由终端主动上传，终端业务配置与服务器端终端登记双向同步。
5. **教师终端必须有显式标识**：教师终端的判断逻辑不是依赖硬件型号，而是依赖终端用途标记与所加载桌面能力。

<a id="scenarios"></a>
## 4. 关键业务场景

### 4.1 单教室标准交付

1. 安装工程师选择一台终端作为临时母机。
2. 通过安装 U 盘制作教学桌面，并在需要时预留一版未加装额外软件的桌面副本。
3. 按物理位置核对同教室学生终端并批量部署。
4. 单独完成教师终端部署；该动作可与学生终端分发并行进行。
5. 导出教室终端清单。
6. 完成本教室交付测试与验收。

### 4.2 兼容路径：先创建教室，后制桌面

1. 一批终端已预装云桌面管理系统，但尚未制作任何桌面。
2. 安装工程师通过临时母机汇总终端信息并导出教室终端清单。
3. 服务器导入清单，先创建教室并登记终端信息。
4. 后续任一终端制作桌面并上传后，服务器再继续向目标终端下发桌面。

### 4.3 多教室交付路径

1. 基础路径是每间新教室都可独立重复单教室部署流程，不依赖服务器能力。
2. 增强路径是在首间教室完成桌面制作后，通过服务器辅助向其他教室批量下发。
3. 在没有服务器且确有跨教室复用诉求时，可通过离线桌面打包携带完成跨教室部署。

### 4.4 交付测试与验收

1. 单教室完成安装后，临时母机当前教室界面和服务器平台都应能够发起并查看硬件检测、硬盘测速和基础网络测试。
2. 硬件基线比对与外设状态检查默认自动执行，并在管理界面中持续可见。
3. 广播风暴测试属于跨教室网络测试，必须明确选择至少任意两间已交付教室共同参与。
4. 测试结果应支持当前教室查看、服务器平台查看和结果导出；终端接入服务器后，本地测试与告警结果自动同步到平台。

### 4.5 日常运维与桌面更新

1. 机房管理员在单教室内响应加装软件等常见需求，并完成本地分发。
2. 单教室验证通过后，可由服务器继续向其他教室批量下发。
3. 日常课堂中由教师终端承担开关机主入口。
4. 常见运维动作应优先设计为机房管理员在远程指导下即可掌握。

### 4.6 考试环境

1. 考试桌面可在单教室初次交付阶段一并制作，也可在后续按需补做。
2. 考试桌面与教学桌面可以并存，未启用时可隐藏其入口。
3. 将考试桌面批量下发到目标教室终端。
4. 设置唯一启动项与还原策略。
5. 对不接受 VHD 或类似承载方式的考试软件，在二期提供物理分区部署兼容路径。

<a id="functional-requirements"></a>
## 5. 功能需求

### 5.1 交付、登记与多教室复用

#### FR-01 母机教学桌面制作
- **目标**：在单教室内制作教学桌面，并在需要时预留一版未加装额外软件的桌面副本供后续继续派生。
- **执行主体**：安装工程师。
- **主流程**：
  1. 选择一台终端作为临时母机，插入安装 U 盘。
  2. 如终端尚未安装云桌面管理系统，则通过安装 U 盘完成底层安装。
  3. 在云桌面管理系统中点击“制作桌面”，读取安装 U 盘中的基础镜像。
  4. 生成初始桌面后，可直接编辑，也可先保留一版未加装额外软件的桌面副本，再继续制作教学桌面。
  5. 如需独立数据盘，应在制作阶段确认数据盘设置。
  6. 完成教学软件安装、激活与个性化设置后，重启返回底层系统。
- **系统要求**：
  1. 支持 Windows 10/11、麒麟 OS、统信 UOS 等常见基础镜像。
  2. 教学桌面支持禁用部分快捷键与禁止自动更新等教学约束。
  3. 页面里直接展示桌面名称、时间和备注，供工程师识别自己预留的基础桌面。
- **优先级**：1 期。

#### FR-02 单教室终端批量部署
- **目标**：将一个或多个待部署桌面按当前页确认结果批量下发到同教室终端，并完成网络身份与用途写入。
- **执行主体**：安装工程师。
- **主流程**：
  1. 在临时母机中点击“批量部署”，系统扫描同网段内已上电的可部署终端，并提供当前教室终端布局或终端列表核对界面。
  2. 结合物理位置和终端当前字段核对终端，支持按当前 IP、当前机器名、当前工作 MAC 等字段排序勾选，并在同一轮里设置终端用途、座位号、IP 段、机器名等写入规则。
  3. 在确认阶段核对并微调这轮勾选终端、保持现状终端、用途、座位号、IP 和计算机名。
  4. 选择一个或多个待部署桌面；如需补做其中某个桌面，可新建、复制或进入编辑该桌面后继续下发。
  5. 系统按确认结果写入 IP 地址、计算机名、座位号和终端用途；这轮未纳入执行的终端保持现状。
  6. 若已知服务器地址，可一并写入，并同时设置默认启动桌面与还原模式。
- **系统要求**：
  1. 执行前必须提供完整逐台预览，支持逐台微调或排除。
  2. 终端选择至少支持全选、清空和按字段排序后逐台勾选。
  3. 默认启动桌面与还原模式应允许按教室要求调整。
  4. 单教室新机部署默认整体覆盖纳入执行终端的现有桌面内容；若需保留，必须显式选择。
  5. 部署过程支持断点续传。
  6. 分发完成后应停留在传输结果页，便于工程师现场肉眼核对每台终端的完成状态与卡住位置。
  7. 分发结果应清楚区分已完成、未完成和失败终端，便于后续继续执行和导出。
- **优先级**：1 期。

#### FR-03 教师终端部署
- **目标**：完成教师终端桌面部署，并在系统中明确标记其教师用途。
- **执行主体**：安装工程师。
- **关键要求**：
  1. 教师终端沿用普通终端模型，不单独引入不同硬件体系。
  2. 教师终端通常会补充更多授课软件和现场设置，因此保留单独部署流程。
  3. 教师终端在终端业务配置与服务器端终端登记中都必须具备“教师机”标记。
  4. 教师终端完成部署后，教师桌面应至少具备教室开关机能力和终端在线状态查看能力。
  5. 教师终端部署可以与单教室终端批量部署并行进行，需工程师前往目标终端本机完成，不必在母机页远程展开详细过程。
  6. 如果需要制作或复制教师桌面，应沿用单教室桌面准备规则，但执行位置在教师终端本机。
- **优先级**：1 期。

#### FR-04 教室终端清单导出
- **目标**：导出服务器创建教室和登记终端所需的标准桥梁文件，同时包含本教室桌面包。
- **执行主体**：安装工程师。
- **字段要求**：
  1. 教室名称、座位号、终端用途标记。
  2. IP 地址、计算机名、MAC 地址列表、当前工作 MAC。
  3. CPU、内存、硬盘型号与硬盘出厂唯一 SN 码。
  4. 当前工作 MAC 应由系统自动识别当前实际联网使用的 MAC，并高亮展示，便于现场核对。
- **说明**：正常情况下教室终端清单在桌面部署完成后导出；对于尚未制作桌面的预装终端场景，系统也应兼容直接导出。
- **优先级**：1 期。

#### FR-05 服务器部署与账号初始化
- **目标**：完成服务器管理平台的安装、激活和账号体系初始化。
- **执行主体**：安装工程师。
- **主流程**：
  1. 在服务器执行自动部署脚本完成平台安装。
  2. 首次访问平台时创建系统管理员账号。
  3. 按项目需要创建校区子管理员账号。
  4. 导入授权许可证书并设置服务器 IP 地址或域名。
- **优先级**：1 期。

#### FR-06 教室清单导入与教室创建
- **目标**：将单教室导出的清单导入服务器，创建教室布局并登记终端信息。
- **执行主体**：安装工程师。
- **关键要求**：
  1. 批量创建教室并生成座位布局。
  2. 建立服务器端终端登记，保存 MAC 地址列表与当前工作 MAC。
  3. 保留终端用途标记，使教师终端在布局图中与学生终端可区分。
  4. 导入后立即开始对齐在线状态。
- **优先级**：1 期。

#### FR-07 多教室按单教室流程重复交付
- **目标**：在多教室交付场景下，让每间新教室都能独立重复单教室部署流程。
- **执行主体**：安装工程师。
- **关键要求**：
  1. 多教室交付的基础路径不依赖服务器能力。
  2. 每间新教室都可独立完成桌面制作、批量部署、教师终端部署、清单导出与交付验证。
  3. 服务器辅助复用属于效率增强能力，而不是基础交付前提。
- **优先级**：1 期。

#### FR-08 服务器辅助跨教室桌面复用
- **目标**：在有服务器条件下，减少多教室交付时学生终端与教师终端的重复人工安装成本。
- **执行主体**：安装工程师。
- **关键要求**：
  1. 首间教室完成桌面制作并上传后，服务器可向其他教室学生终端下发教学桌面。
  2. 首间教室完成教师桌面并上传后，服务器可向其他教师终端下发教师桌面。
  3. 是否采用该路径，应结合现场网络传输效率和人工部署效率综合判断。
- **优先级**：1 期。

#### FR-09 离线桌面打包携带
- **目标**：在没有服务器的多教室交付场景下，提高桌面的跨教室复用效率。
- **说明**：离线复制与服务器辅助复用并列为多教室交付主路径，默认支持导出桌面包并在目标教室导入继续部署。
- **优先级**：1 期。

#### FR-10 交付结果校验与当前教室异常查看
- **目标**：在交付现场完成结果核对，并让当前教室异常在最近入口可见。
- **执行主体**：安装工程师。
- **关键要求**：
  1. 在有服务器时，支持通过平台核对教室布局、座位顺序、在线状态和 IP 规划。
  2. 在无服务器时，至少应支持通过临时母机当前教室界面完成本教室结果核对。
  3. 当前教室异常提示应在母机或局域网控制入口可直接看到，避免必须回到服务器端才发现问题。
- **优先级**：1 期。

#### FR-11 交付测试流程与结果组织
- **目标**：明确硬件与网络测试在单教室、多教室、临时母机当前教室界面和服务器平台之间的执行、查看与导出方式。
- **执行主体**：安装工程师、机房管理员。
- **关键要求**：
  1. 单教室部署完成后，临时母机当前教室界面和服务器平台都应能够发起并查看硬件检测、硬盘测速或坏道风险识别及基础网络测试，不要求必须先在终端执行再上传结果。
  2. 硬件基线比对与外设状态检查应默认自动采集、自动判断并在管理界面中展示。
  3. 硬盘测速或坏道检测可由临时母机当前教室界面或服务器平台主动发起，结果自动进入测试视图与告警视图。
  4. 临时母机当前教室界面只展示当前教室可直接执行的测试项；跨教室测试必须单独选择参与范围。
  5. 广播风暴测试必须显式要求至少任意两间已交付教室共同参与，且测试范围限定在当前账号可见校区内。
  6. 临时母机当前教室界面支持导出单教室测试结果；服务器平台支持导出单教室结果或多教室汇总结果。
  7. 测试结果应支持当前教室查看、服务器平台查看和结果导出；终端接入服务器后，本地测试与告警结果自动同步到平台。
- **优先级**：1 期。

### 5.2 平台资产视图与告警

#### FR-12 平台总览首页
- **目标**：提供校区级运行概览。
- **主要使用者**：机房管理员；技服工程师在远程支持时可辅助查看。
- **使用方式**：通过系统管理员账号或校区子管理员账号登录服务器管理平台查看。
- **展示内容**：
  1. 首页至少应有"校区规模"区域，展示教室总数、已创建教室数、登记终端总量以及教师终端和学生终端规模。
  2. 首页至少应有“终端运行概况”区域，展示在线终端数、离线终端数、异常终端数、整体在线率以及需要优先关注的异常教室。
  3. 首页至少应有“服务器运行概况”区域，展示 CPU、内存、存储、授权使用情况以及当前任务占用状态。
  4. 首页至少应有“任务进度”区域，展示桌面上传、跨教室桌面下发、一键替换和一键重置等长任务的进行中、阻塞和失败情况。
  5. 首页至少应有“异常与风险”区域，展示当前未处理告警数、按类型分布、按教室分布以及高频故障终端或高风险教室。
  6. 首页统计卡片和列表应支持跳转到对应教室、终端、告警或任务明细。
- **优先级**：1 期。

#### FR-13 教室布局与终端详情
- **目标**：让平台能够以业务友好的方式展示每个终端，尤其是教师终端与多网卡信息。
- **主要使用者**：机房管理员。
- **关键要求**：
  1. 教师终端在布局图中必须有明确视觉标识，不应与学生终端完全等价显示。
  2. 终端详情页中，当前工作 MAC 应高亮展示；其余 MAC 地址以列表或折叠方式展示，当前工作 MAC 由系统自动识别当前实际联网使用的 MAC，便于现场核对。
  3. 终端详情中至少展示教室、座位、终端用途、在线状态、当前桌面、最近心跳时间等运行识别信息。
  4. 终端详情中至少展示 IP、计算机名、当前工作 MAC、MAC 地址列表等身份与网络信息。
  5. 终端详情中至少展示 CPU、内存、硬盘型号、硬盘 SN、网卡信息以及显示器、键盘、鼠标、耳机、摄像头等关键硬件与外设摘要。
  6. 终端详情中至少展示当前告警、最近测试结果和最近同步或上报时间等运维辅助信息。
- **优先级**：1 期。

#### FR-14 告警展示
- **目标**：将服务器和终端侧异常统一可见化。
- **主要使用者**：机房管理员；技服工程师在远程支持时可辅助查看。
- **关键要求**：
  1. 一期重点是展示告警。
  2. 终端侧硬件基线比对与外设状态检查应默认自动执行，并在有管理界面查看时直接展示。
  3. 服务器平台应展示当前校区异常概况与教室级异常明细；终端接入服务器后，本地告警自动同步到平台。
  4. 母机或临时母机在现场操作时，应能看到当前教室的异常提示。
  5. 告警至少覆盖硬件不一致、外设未连接、硬盘掉速、坏道、终端离线、服务器资源异常等情况；具体告警项允许按产品设计补充。
  6. 告警列表应支持按教室、类型、时间和处理状态筛选，并支持从首页跳转到教室或终端详情。
  7. 告警列表应支持按等级、类型、时间和教室快速筛选或排序，帮助优先定位重点异常。
- **优先级**：1 期。

#### FR-15 告警处置
- **目标**：为合理硬件替换或已确认异常提供消警能力。
- **主要使用者**：机房管理员。
- **说明**：
  1. 二期提供“接受”“忽略”等处置动作。
  2. “接受”表示当前状态被认定为合理，并可更新基线。
  3. “忽略”表示保持原始基线，但不再重复提示该条告警。
- **优先级**：2 期。

### 5.3 日常教学运行、加装软件与桌面下发

#### FR-16 教师终端一键开关机
- **目标**：支持教师终端和临时母机对学生终端执行全部开机、全部关机。
- **执行主体**：教师、机房管理员。
- **关键要求**：
  1. 日常上课场景下，由教师终端承担主要入口。
  2. 交付、验收或排障场景下，临时母机也可执行相同的开关机控制。
  3. 当前需求仅覆盖开机和关机，不扩展为更重的复杂群控套件。
- **优先级**：1 期。

#### FR-17 单教室加装软件
- **目标**：支持响应教师或学校的临时加装软件需求，并优先在单教室内通过本地路径完成快速分发。
- **执行主体**：机房管理员。
- **主流程**：
  1. 选择任一终端作为临时母机，进入编辑桌面。
  2. 完成软件安装、激活、验证后，返回云桌面管理系统。
  3. 系统合并本次桌面修改结果。
  4. 先向同教室其他终端执行本地批量部署。
  5. 再在后台将桌面上传到服务器。
- **关键要求**：后台上传进度必须对用户可见。
- **优先级**：1 期。

#### FR-18 跨教室桌面下发
- **目标**：在单教室完成加装软件或其他桌面调整后，将已验证桌面继续下发到其他教室。
- **执行主体**：机房管理员。
- **关键要求**：
  1. 对用户而言，下发对象是“桌面”。
  2. 下发入口应围绕“来源桌面 + 目标教室”组织，而不是暴露孤立的底层子功能。
  3. 对系统内部而言，下发过程同时处理基础镜像与快照的按需同步。
  4. 上传未完成时，平台必须展示进度且不允许误下发。
- **优先级**：1 期。

### 5.4 故障恢复

#### FR-19 一键替换
- **目标**：在原工位硬件损坏时，让新终端快速接管原有身份和桌面。
- **执行主体**：机房管理员。
- **主流程**：
  1. 将新终端接入原座位的网线与电源并开机。
  2. 在底层系统中点击“一键替换”，确认或修正当前服务器地址与本机网络信息。
  3. 选择需替换的教室与座位。
  4. 系统下发原座位的业务配置和桌面资产。
  5. 新终端完成上报后，服务器更新其 MAC 地址列表、当前工作 MAC 与硬件信息。
- **优先级**：1 期。

#### FR-20 一键重置
- **目标**：在终端遭遇蓝屏、中毒、异常卡顿等软件故障时，快速恢复可用环境。
- **执行主体**：机房管理员。
- **关键要求**：
  1. 一键重置依赖服务器作为权威恢复源，从服务器重新拉取当前终端已登记的桌面与配置并重刷本机。
  2. 如果服务器不可达或服务器地址未配置，不应直接执行重置，而应允许在当前流程内修正本机 IP、网关、DNS 和服务器地址后继续。
  3. 本地已有镜像和快照属于运行所需本地数据，不单独定义为可依赖的缓存恢复源。
  4. 整个恢复过程可视化展示进度。
- **优先级**：1 期。

### 5.5 校园网改造

#### FR-21 终端 IP 批量变更
- **目标**：在学校网段变更时，按座位顺序批量重写终端 IP 配置。
- **执行主体**：机房管理员。
- **入口**：
  1. 单教室入口：临时母机中的“批量修改 IP 地址”。
  2. 服务器入口：服务器平台中的“批量修改 IP 地址”。
- **关键要求**：
  1. 修改后终端业务配置与服务器端终端登记保持一致。
  2. 必要时可由技服工程师通过远程方式提供指导。
- **优先级**：1 期。

#### FR-22 服务器地址变更
- **目标**：在服务器 IP 地址或域名变更后，尽量避免终端失联。
- **执行主体**：机房管理员。
- **关键要求**：
  1. 必须遵循“先通知终端，后修改服务器本体”的顺序。
  2. 支持服务器端向在线终端推送新地址。
  3. 支持对漏通知或离线终端前往目标终端单独修改网络与服务器配置作为后续处理。
  4. 若启用 HTTPS，地址变更后支持导入新 SSL 证书。
  5. 必要时可由技服工程师通过远程方式提供指导。
- **优先级**：1 期。

#### FR-22A 单终端网络与服务器修正
- **目标**：在单终端需要恢复、替换、处理失联或首次接入时，允许本机直接修正网络与服务器配置。
- **执行主体**：机房管理员。
- **关键要求**：
  1. 支持在单终端本地页查看并修改本机 IP、掩码、网关、DNS 和服务器地址。
  2. 修改后应立即测试与服务器连通性。
  3. 一键替换和一键重置流程应能内联复用这组字段，不应把用户打断到独立功能孤岛。
- **优先级**：1 期。

#### FR-22B 本机信息查看与名称修正
- **目标**：让机房管理员在单终端本地页快速核对本机识别信息，并在必要时修正本机名称。
- **执行主体**：机房管理员。
- **关键要求**：
  1. 至少展示本机名称、当前工作 MAC、MAC 地址列表、关键硬件摘要和当前登记状态。
  2. 支持在需要时修正本机名称。
  3. 教室、座位和终端用途仍以部署或维护流程写入为主，不在这里承担教室级范围调整。
- **优先级**：1 期。

### 5.6 考试环境

#### FR-23 考试桌面制作
- **目标**：在现有桌面体系上追加或制作独立考试桌面。
- **执行主体**：技服工程师。
- **关键要求**：
  1. 支持基于已有桌面副本继续制作考试桌面，也支持按需新做考试桌面。
  2. 支持进入编辑桌面完成考试软件安装和验证。
  3. 考试桌面可在单教室初次交付阶段一并制作，也可在后续按需补做。
  4. 考试桌面与教学桌面可以并存，未启用时可隐藏其入口。
  5. 页面里直接展示桌面名称、时间和备注，供工程师识别自己预留的基础桌面。
- **优先级**：1 期。

#### FR-24 考试桌面批量部署
- **目标**：将考试桌面批量下发到目标教室终端。
- **执行主体**：技服工程师。
- **关键要求**：
  1. 支持在母机或服务器辅助下将考试桌面批量下发到目标终端。
  2. 教师终端和学生终端都应按考试要求配置到位。
- **优先级**：1 期。

#### FR-25 考试启动项与还原设置
- **目标**：在考试前将终端设置为默认进入考试桌面，并设定正确的入口可见性与还原策略。
- **执行主体**：技服工程师。
- **关键要求**：
  1. 将考试桌面设置为默认进入的唯一启动项，并隐藏其他桌面入口。
  2. 根据考试要求设置还原模式，通常为“系统盘还原、数据盘保留”。
  3. 若后续启用物理分区式考试桌面，启动项必须直接指向对应物理分区桌面。
  4. 考后可沿用原批量设置入口恢复考前默认启动策略。
  5. 不应要求考后逐台人工回切。
- **优先级**：1 期。

#### FR-26 物理分区式考试桌面
- **目标**：兼容少量不接受 VHD 或类似虚拟化承载方式的敏感考试软件。
- **执行主体**：技服工程师。
- **关键要求**：
  1. 二期支持将考试桌面部署到真实物理分区。
  2. 在业务操作层面，尽量沿用“制作桌面、批量部署、设置启动项”的统一心智模型。
  3. BIOS 引导策略需要能够把该物理分区桌面设置为加载目标。
- **优先级**：2 期。

### 5.7 账号与日志

#### FR-27 校区子管理员隔离
- **目标**：在多校区场景下实现同能力、不同可见范围的账号隔离。
- **系统要求**：
  1. 系统管理员账号可见全部校区。
  2. 校区子管理员账号仅可见所属校区的教室和终端。
  3. 两者的基础能力集尽量保持一致，降低培训与使用成本。
- **优先级**：2 期。

#### FR-28 操作日志
- **目标**：支持问题追踪与责任溯源。
- **系统要求**：
  1. 服务器平台记录管理员账号的关键操作行为。
  2. 终端本地记录关键操作行为，包括桌面上传、替换、重置、批量部署、教室开关机等动作。
  3. 日志字段至少包含操作人、账号、时间、操作对象、动作内容、执行结果。
- **优先级**：2 期。
