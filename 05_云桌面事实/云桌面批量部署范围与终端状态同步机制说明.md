## 1. 为什么这份结论会同时约束母机页和受控任务屏

同一轮单教室任务开始后，母机页和受控终端任务屏看到的是同一批对象的不同切面。

母机页要看整间教室当前有多少终端、这轮到底勾选了哪些终端、谁已经完成、谁还没完成；受控终端任务屏只关心本机是否被纳入、本机当前执行到哪一步、本机是不是保持现状。

因此这份文档不是单独解释某一页，而是先把这几种状态之间的关系固定下来，避免母机页和受控任务屏各自写出一套不同口径。

## 2. 三类对象必须分开表达

### 2.1 教室受管集合

1. 这是母机接管成功后，当前由本机作为母机管理的整批终端。
2. 它服务于母机查看整间教室、补录终端、导出清单和后续维护。
3. 讲台位终端、教师终端和待单独处理终端都可以留在这个集合里。

### 2.2 这轮勾选终端

1. 这是当前这一次任务真正要处理的终端子集。
2. 它来自教室受管集合，但不等于整个教室受管集合。
3. 某些终端可以留在教室受管集合里，却不参加这一轮动作，例如教师终端或准备后续单独处理的终端。

### 2.3 这轮保持现状的终端

1. 这类终端仍然属于当前教室。
2. 它们只是这一次没有参加当前任务。
3. 页面上应把它们显示为“本次未纳入执行”或“待单独处理”，而不是掉线或失败。
4. 在逐台确认层里，“保持现状”不应只给一个抽象标签；还应回显当前被保留的用途、座位号、机器名、IP 等已有内容，并明确本次不修改。

### 2.4 这轮不参与的终端只表达“保持现状”

1. 某些终端这轮不参与，只表示它们仍在当前教室受管集合里，但本次不执行。
2. 确认页和结果页应同时写清“本次不修改”和“沿用了什么当前内容”。
3. 批量任务正文不额外展开教师终端、讲台位终端或其他个别终端的后续补路；后续如需单机处理，再进入对应本机入口或故障处理流程。

## 3. 批量设置和真实执行之间必须有逐台确认层

1. 批量设置先决定这轮勾选哪些终端，以及按什么顺序处理。
2. 逐台确认层负责把座位号、IP、计算机名、终端用途和是否纳入执行逐台展示出来。
3. 现场工程师要在这一层处理例外项，例如教师终端排除、单机参数改单机值、局部跳过。
4. 只有在逐台确认后，才进入真实执行。

这不是额外加的一步，而是批量任务在真实现场里必须有的可视化校验层。维护和考试流程在后续页面里也应沿用这条结论。

这里还要固定一条容易漏掉的事实：批量页里使用的机器名、座位号、终端用途、网络配置、服务器地址、桌面集合、默认启动项和还原方式，并不是“只存在于批量流程里的字段”。它们在单机场景下也应能分别通过 TERM-01 的 [本机信息]、[网络与服务器]、[本机桌面] 逐项处理；区别只在整批写入还是单台处理。

### 3.1 TERM-03 上的教室级动作前提要按动作拆开判断

1. TERM-03 上至少同时存在单教室部署、教室维护、考试场景、补录终端、导出清单这些动作；页面要分别表达它们各自的成立条件和下一步入口。
2. 单教室部署遵循“先行设置当前母机，再行单教室终端批量部署”的基线顺序。点击 [单教室部署] 后，TERM-04 先完成当前母机自己的机器名、IP 与网络配置、座位号、终端用途、本机桌面准备，再继续终端范围、批量规则、逐台预览和分发阶段。
3. 教室维护要按维护类型判断。桌面更新类维护依赖当前母机已经准备好的本机桌面；TERM-05 的批量步骤只选择这些已有桌面。如需继续准备桌面，则在桌面选择步骤点击 [修改本机桌面]，进入当前母机的本机桌面处理，可新建桌面、复制已有桌面，或进入桌面编辑；完成这次桌面处理并再次重启后，系统回到原步骤继续维护。批量改终端 IP、批量改服务器地址只要求当前母机还能和受管终端通信，不要求当前母机已有桌面。
4. 考试场景依赖考试桌面准备；考试相关桌面也应先在当前母机本机桌面里准备好，TERM-07 再选择已有桌面并继续批量写考试启动设置和考后恢复。如需继续准备桌面，同样由桌面选择步骤里的 [修改本机桌面] 发起；在那里可新建桌面、复制已有桌面，或进入桌面编辑；完成这次桌面处理并再次重启后回到原步骤。
5. 补录终端和导出清单继续按当前教室识别结果与字段完整性判断，不跟随上述三条批量流程共用同一套门槛。

## 4. 母机和受控终端看到的是不同信息

TERM-03 只负责当前教室工作面；TERM-04、TERM-05、TERM-07 才负责各自任务的设置、执行和结果停留。

因此：

1. 当前没有教室级任务时，TERM-03 默认先看当前教室终端状态、异常和最近测试。
2. 一旦用户已经进入 TERM-04、TERM-05、TERM-07，这轮勾选多少、保持现状多少、谁执行中、谁失败，都留在对应任务页表达。
3. 软件重进时，如果当前任务上下文仍在，应直接回到对应任务页，而不是先回 TERM-03。
4. TERM-03 不再定义任务锁定态、任务返回态或收口态。

### 4.1 母机在 TERM-03 和 TERM-04、TERM-05、TERM-07 上分别看到什么

1. 在 TERM-03 上，母机先看当前教室共有多少终端、当前异常怎样、最近测试怎样，以及当前还能发起哪些教室级动作。
2. 在 TERM-04、TERM-05、TERM-07 上，母机才看这轮勾选了多少台、保持现状多少台、哪些终端执行中、哪些终端已完成、哪些终端失败。
3. 任务页里的“保持现状”要同时说明“本次不修改”和“沿用了什么当前内容”，而不是只留一个抽象状态词。

### 4.2 受控终端在 TERM-01 上看到什么

1. 当前母机是谁。
2. 本机是否纳入这轮执行。
3. 本机计划参数是什么，例如计划 IP、计划座位、计划默认启动项和计划还原方式。
4. 本机当前执行到哪一阶段。

### 4.3 这轮保持现状的终端在 TERM-01 上看到什么

1. 当前仍属于哪间教室的受管终端。
2. 本次任务是什么。
3. 本机这次保持现状，没有参与执行。

## 5. 终端用途和写入字段的表达方式

1. 终端用途是可扩展字段，不应在页面结构里被写死成只有学生和教师两类。
2. 批量设置阶段可以先生成一版默认用途值。
3. 预览阶段允许逐台调整用途、保留现状或排除执行。
4. 服务器地址可以作为部署字段之一写入终端，但它不是单教室部署成立与否的前提。
5. 批量设置里出现的机器名、座位号、终端用途、网络配置、服务器地址、桌面集合、默认启动项和还原方式，在单机场景下也应有一一对应的本机处理入口；批量和单机处理的是同一组业务字段，不是两套含义不同的配置。

## 6. 教室终端清单导出依赖什么

1. 导出依赖的是当前教室本地已经识别到的终端信息和本地部署结果。
2. 导出不以服务器是否存在、是否已创建教室、是否已完成上传为前提。
3. 如果仍有终端关键字段缺失、用途未确认或本次任务未完成，页面应提示“导出内容可能不完整”。
4. 提示的目标是帮助工程师判断当前要不要导出，而不是强行阻塞导出。

## 7. 与服务器的关系

1. 单教室部署可以全程不开服务器。
2. 终端如果已经写入正确服务器地址，后续在服务器存在、已创建教室并登记终端且网络连通时，可以继续自动上传运行状态和桌面资产。
3. 这些上传行为属于后续统一纳管能力，不反向决定当前这次本地批量部署是否算完成。
4. 终端与服务器之间的按需同步细节，继续以 [云桌面终端与服务器按需同步说明.md](./云桌面终端与服务器按需同步说明.md) 为准。
