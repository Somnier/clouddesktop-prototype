# 云桌面原型设计技术背景说明

> 文档角色：记录当前页面设计中已默认成立的技术假设。
> 作用：原始需求清单（01）不包含技术方案文档，页面设计过程中隐含引用了多项技术能力。本文档将这些假设显式化，供后续原型实现和技术验证时对照。

## 1. 本文档的定位

页面设计阶段关注"用户看到什么、能做什么"，但设计中有些功能只有在特定技术能力可行时才成立。这些技术假设散落在各页面文档中，难以逐页核查。本文档将它们集中归纳，明确"设计依赖了哪些技术前提"。

如果后续技术验证发现某项能力不可行，应回到对应页面调整设计方案，并同步更新本文档。

## 2. 终端运行环境

| 技术假设 | 涉及页面 | 设计依赖说明 |
|---|---|---|
| 终端管理软件作为本地桌面应用运行（Electron 或同类方案） | TERM-01 ~ TERM-07 | 全部终端侧页面均依赖管理软件能以 Native 应用形式运行，具备本地文件读写、进程管理和 UI 锁定能力 |
| 受控终端任务屏能锁定系统 UI | TERM-01、事实：任务屏机制 | 受控终端在执行批量任务期间，管理软件需拦截用户进入原生桌面，仅展示任务状态 |
| 跨重启任务上下文保持 | TERM-04、TERM-05、TERM-07 | "修改本机桌面"流程：点击后终端重启进入桌面编辑环境，编辑完成后再重启回到管理系统原步骤。系统需跨重启保存当前任务进度并恢复 |

## 3. 磁盘与桌面技术

| 技术假设 | 涉及页面 | 设计依赖说明 |
|---|---|---|
| 桌面基于磁盘镜像（VHD 或同类）管理，支持分层与快照 | TERM-03、TERM-04、WEB-03 | 桌面制作、克隆、编辑、资产管理均依赖镜像分层技术（母盘 + 增量快照） |
| 离线桌面包可导出导入 | TERM-04、WEB-03 | 无服务器环境下，母机通过 .deskpkg 文件完成本地部署；平台侧支持桌面包导入导出 |
| 磁盘可被远程重写和恢复 | TERM-06 | "一键重置"需要从网络拉取完整镜像并覆盖本地磁盘分区 |

## 4. 网络与批量操作

| 技术假设 | 涉及页面 | 设计依赖说明 |
|---|---|---|
| 母机可局域网扫描发现在线终端 | TERM-02、事实：母机接管机制 | 接管教室时，母机需自动扫描当前网段内的终端并读取其教室绑定信息 |
| 母机可远程改写受控终端的网络配置 | TERM-04、TERM-05 | 批量部署和维护流程中，母机需远程配置受控终端的静态 IP、子网掩码、网关和 DNS |
| 大文件批量分发具备加速手段（组播或同类） | TERM-04 | 单教室部署需向数十台终端下发 GB 级桌面镜像，设计上假设存在组播或 P2P 分发机制以避免单播带宽瓶颈 |
| 平台侧可跨教室发起广播隔离测试 | WEB-02 | 多教室测试工具栏的"广播测试"假设服务器能向多间教室同时下发测试指令，并隔离各教室结果 |

## 5. 数据同步与通信

| 技术假设 | 涉及页面 | 设计依赖说明 |
|---|---|---|
| 母机与受控终端之间存在心跳和指令通道 | TERM-01、事实：任务屏机制 | 受控终端任务屏需实时显示母机连接状态和任务进度，依赖持续心跳监测 |
| 终端资产可后台异步上传服务器 | 事实：按需同步 | 本地部署完成后，GB 级桌面资产在后台利用余量带宽异步同步至服务器 |
| 终端状态变更可推送至平台 | WEB-01、WEB-02、事实：按需同步 | 平台总览和教室管理的实时指标依赖终端主动上报或服务器定期拉取 |

## 6. 硬件识别

| 技术假设 | 涉及页面 | 设计依赖说明 |
|---|---|---|
| 终端可自动读取 MAC、硬盘 SN、CPU/内存/磁盘型号 | TERM-01、WEB-02A | 终端首页和平台终端详情展示的硬件信息依赖管理软件直接读取系统 API |
| MAC 地址作为终端唯一标识字段 | TERM-06、事实：母机接管机制 | 故障处理的"一键替换"通过新旧 MAC 关联座位，换机识别以 MAC 为最高优先级 |
