## 1. 为什么需要定义三层关系

终端侧、部署流程和平台侧都涉及"桌面"这个概念，但三者操作的粒度不同。如果不把底层数据模型的层级关系固定下来，后续页面很容易把"桌面""镜像""快照"混用，或者把平台侧的资产删除和终端侧的桌面删除混成同一件事。

## 2. 三层各自是什么

### 2.1 基础镜像（Image）

1. 基础镜像是底层操作系统安装的物理磁盘映像文件。
2. 一个基础镜像对应一个操作系统版本，例如"Windows 11 23H2"。
3. 基础镜像通过三种途径进入云桌面管理系统：从外部基础镜像文件（基础镜像包）导入、从外部桌面文件（桌面包）导入，或由终端接收母机传输的桌面后产生。
4. 基础镜像不直接面向终端用户操作，但它是所有上层桌面的共同底座。

### 2.2 快照（Snapshot）

1. 快照是基于基础镜像之上的增量层，记录某一时刻的磁盘差异。
2. 一个基础镜像可以派生多个快照，形成快照树。
3. 通过基础镜像创建桌面时，会在基础镜像下创建一个快照，永远不会直接对基础镜像做修改。
4. 每次桌面编辑完成后，更新的是当前桌面对应的快照，不是基础镜像本身。
5. 复制桌面时创建新快照，新快照的父节点指向源桌面的快照。
6. 快照的存在使得同一基础镜像上可以同时维护多个桌面而不互相干扰。

### 2.3 桌面（Desktop）

1. 桌面是面向终端用户和管理员的可操作对象。
2. 每个桌面指向一个快照，通过快照间接关联基础镜像。
3. 桌面包含用户可见的业务属性：名称、用途类型、版本、数据盘配置、还原方式、可见性和同步状态。
4. 终端侧的所有桌面操作（新建、复制、编辑、删除、设为默认、隐藏）都在桌面层进行。

## 3. 三层关系图

```
基础镜像 (Image)
  ├── 快照 A (Snapshot) ── 桌面 1: 公共教学桌面
  ├── 快照 B (Snapshot) ── 桌面 2: 考试桌面
  └── 快照 C (Snapshot) ── 桌面 3: 软件开发实训桌面
```

一个教室可以有多个基础镜像，每个基础镜像下有各自的快照树，每个叶子快照对应一个桌面。

## 4. 不同页面在哪一层操作

| 操作场景 | 操作层级 | 说明 |
|---|---|---|
| TERM-01 新建桌面（从镜像） | Image → Snapshot → Desktop | 导入基础镜像，自动创建初始快照，生成桌面 |
| TERM-01 新建桌面（从桌面包） | Image → Snapshot → Desktop | 桌面包内含基础镜像、快照和桌面业务属性，一次性导入三层 |
| TERM-01 复制桌面 | Snapshot → Desktop | 从源桌面的快照派生新快照，创建新桌面 |
| TERM-01 编辑桌面 | Snapshot（原地更新） | 重启进入桌面编辑环境，修改的差异写回当前快照 |
| TERM-01 删除桌面 | Desktop + Snapshot | 删除桌面及其关联的快照；如果该快照是所属基础镜像下的唯一快照，级联删除基础镜像 |
| TERM-04/05 批量部署 | Desktop（集合） | 选择多个桌面下发到受控终端，每台终端获得桌面的副本 |
| 终端→服务器上传 | Snapshot + Image | 上传时传输的是快照增量和所依赖的基础镜像 |
| WEB-03 桌面资产管理 | Image + Desktop（聚合视角） | 平台侧按基础镜像聚合，展示每个镜像下有哪些桌面、被多少教室引用 |
| WEB-03 删除资产 | Desktop + Snapshot + Image | 删除无引用的桌面资产时级联清理孤立快照和基础镜像 |

## 5. 跨端数据口径

1. 终端侧展示的"桌面"列表，是当前终端本地的 Desktop 层对象，包含业务属性和同步状态。
2. 平台侧展示的"桌面资产"，是服务器已经收到的桌面数据在 Image 层的聚合视图，按教室和引用关系分组。
3. 终端侧的"同步状态"（未同步/同步中/已同步）描述的是当前桌面的快照和基础镜像是否已上传到服务器。
4. 平台侧的"桌面引用数"统计的是有多少教室的终端引用了同一镜像下的桌面。

## 6. 桌面包作为跨端交换格式

1. 桌面包（.deskpkg）是包含基础镜像、快照、桌面业务属性和数据盘配置的完整交换单元。
2. 从镜像文件新建桌面时，只导入基础镜像层，数据盘和业务属性需用户在新建过程中设定。
3. 从桌面包导入时，三层数据和业务属性一并还原，适用于跨教室、跨终端的桌面迁移场景。
4. 导出桌面包时，打包当前桌面对应的完整三层数据。
