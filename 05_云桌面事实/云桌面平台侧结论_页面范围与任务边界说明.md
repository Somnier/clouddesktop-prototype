# 云桌面平台侧结论_页面范围与任务边界说明

> 文档角色：平台侧稳定结论之一。
> 作用：把平台侧已经明确的职责范围、数据来源和任务边界放在一处，供后续平台页面设计直接引用。

## 1. 平台侧首先承接什么

平台侧不是把终端本地流程重新搬到 Web 上，而是承接服务器纳管之后的汇总、查看、教室创建与终端登记、跨教室动作和运维入口。

平台侧工作至少包括：

1. 校区、教室、终端的汇总查看。
2. 教室创建、教室布局和终端登记信息查看。
3. 终端详情、当前状态、网络身份、硬件外设、测试和告警查看。
4. 跨教室桌面部署和相关任务反馈。
5. 测试结果、告警结果和服务器运维入口。
6. 桌面资产管理：平台侧支持桌面包的导入、导出，以及删除无引用的桌面资产。

## 2. 平台侧不应该重写什么

平台侧需要看见终端结果，但不应该反过来改写终端本地页面已经确认的判断。

下面几类事情，不适合由平台侧重新定义：

1. 单教室本地部署是否算完成。
2. 某一台受控终端在当前批量任务里是否已完成。
3. 母机本地这轮任务的结果页该怎么停留和怎么继续处理。
4. 单终端故障处理在本机上的入口与流程。

平台可以展示这些结果，但不应替代终端侧对这些结果的本地业务判断。

## 3. 平台侧主要依赖哪些数据

平台侧当前主要依赖 4 类数据：

1. 服务器侧终端登记信息，例如教室、座位、终端用途、MAC 地址列表和当前工作 MAC。
2. 终端上传的运行状态，例如在线状态、当前执行态、资源占用、最近同步时间。
3. 终端上传的桌面资产和桌面相关状态。
4. 测试结果和告警结果。

这里的“平台侧”指 Web 页面与平台用户操作视角；“服务器侧”指服务器本体状态、地址、证书以及服务器侧记录的数据。“配置”用于持久业务字段；“设置”用于当前页面上的选择、输入和本次写入计划。

因此平台页面里必须分清：哪些是服务器已经登记的长期信息，哪些是终端刚刚上传的当前状态，哪些是任务执行中的进度，哪些只是历史结果。

在线状态只代表终端当前是否连通，不等于终端当前空闲；如果平台要表达“正在执行本地任务”或“正在执行平台下发任务”，应另给当前执行态或任务进度字段，不能拿在线状态代替。

## 4. 平台侧页面范围

平台侧至少应把下面几类页面边界说清楚：

1. 平台总览首页：看校区规模、终端运行概况、服务器运行概况、任务进度和异常风险。
2. 教室列表与教室页：看教室布局、当前任务、当前异常、当前测试结果；单教室和跨教室桌面部署、批量修改 IP、网络测试和广播隔离测试均在教室页批量工具栏发起。
3. 终端详情：看终端身份、网络、硬件、外设、告警和最近同步情况。
4. 桌面资产管理：看校区级桌面全貌，支持桌面包的导入、导出和删除无引用桌面。
5. 告警中心：看异常汇总、按教室和类型筛选、排序和后续排查入口。
6. 系统运维：看服务器地址变更、证书、授权管理、校区与账号管理和其他运维动作。

## 5. 平台侧和终端侧怎样配合

平台侧与终端侧之间，不是前后替代关系，而是本地结果与统一纳管结果并存。

更准确地说：

1. 终端侧先完成本地交付、本地部署、本地维护和本地故障处理。
2. 平台侧在服务器上线、教室创建与终端登记完成、网络连通后，继续承接汇总查看和跨教室复用。
3. 某轮本地任务已经完成，不需要等待平台侧同步成功才算成立。
4. 平台侧同步失败，会影响跨教室复用和统一查看，但不反向推翻终端本地已经完成的结果。

## 6. 平台侧后续页面设计最容易错在哪里

平台页面后续展开时，最容易出现四类偏差：

1. 把终端本机当前状态、服务器端登记信息和任务进度混成一条字段。
2. 把终端本地主线动作直接搬成 Web 页面动作，忽略两边操作场景不同。
3. 因为平台侧能看见更多汇总信息，就反向改写终端侧本地完成判断。
4. 跨教室桌面部署从目标范围选择直接跳到执行结果，中间没有按教室预览和终端级核对层。

后续只要平台页面持续避开这四类偏差，平台侧原型就更容易保持清楚的职责边界。
