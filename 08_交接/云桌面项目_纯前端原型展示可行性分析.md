# 云桌面项目纯前端原型展示可行性分析

> 文档日期：2026-03-30
> 文档角色：面向前端原型设计和演示的可行性分析说明。
> 适用范围：用于判断如果当前项目先不进入完整研发，而是先做纯前端原型展示项目，可以做到什么程度、缺什么环节，以及 AI Agent 能提供哪一层级的高保真输出。

## 1. 这份文档讨论的前提

这里讨论的不是完整产品研发，而是更收敛的一种目标：

1. 先做一个纯前端原型展示项目。
2. 目标是把核心页面、主要流程、状态变化和信息层级展示清楚。
3. 这个项目可以服务于内部评审、方案汇报、需求澄清、研发预演或客户演示。

在这个前提下，需要把两个问题拆开：

1. 纯前端原型展示项目是否可做。
2. 纯前端原型展示项目能替代到什么程度，不能替代什么。

## 2. 结论先行

如果目标是做一套高质量的前端交互演示原型，当前输入已经足够开工，而且可以做得比较完整。

但如果目标是希望这个“纯前端原型展示项目”同时替代真实工程验证、真实设备能力验证和真实任务链路验证，那还不够，也不应该这样期待。

更直接地说：

1. 当前输入足够支撑高质量页面原型、流程原型、状态原型和汇报级展示原型。
2. 当前输入不足以让纯前端原型代替终端能力验证、协议验证、后台长任务验证和部署验证。
3. 当前输入可以支撑较高保真的视觉和交互稿，但高保真不等于真实可运行系统。

## 3. 为什么说纯前端原型展示现在就能做

### 3.1 页面结构已经足够稳定

当前 06 和 07 已经把终端端、平台端的页面结构拆清：

1. 终端侧有首页、本机信息、网络与服务器、本机桌面、母机接管、母机工作页、部署、维护、故障处理、考试等完整主线。
2. 平台侧有总览、教室页、终端详情、桌面资产下发、测试与告警、系统运维等完整主线。

对纯前端原型来说，这已经足够建立信息架构、页面路由、演示导航和多场景切换。

### 3.2 页面之间的关系已经足够清楚

纯前端原型最怕的不是样式，而是页面之间没有闭环。当前这套文档已经给出了比较完整的衔接关系：

1. 终端从 TERM-01 进入不同主线。
2. 母机接管后进入 TERM-03，再进入 TERM-04、TERM-05、TERM-07。
3. 受控任务屏与母机页、TERM-06 的互斥边界已经说明。
4. 平台从 WEB-01 可以进入教室、终端、告警、测试和系统运维。

这意味着前端原型可以不只是静态页面集合，而是较完整的“可点击流程样机”。

### 3.3 状态表达已经足够支撑演示级交互

当前事实层已经把很多关键状态拆开，例如：

1. 当前值、配置值、准备写入值。
2. 这轮勾选终端、保持现状终端、当前教室受管集合。
3. 在线状态、执行状态、测试摘要、异常摘要。
4. 本地完成与平台同步失败并行成立。
5. 桌面准备、桌面选择、批量执行、结果停留的分层。

这类状态拆分对真实研发当然还不够，但对演示原型已经非常关键，因为它能让页面展示出真正像产品而不是像海报。

## 4. 纯前端原型展示项目可以做到什么程度

### 4.1 可以做到的内容

如果以 Angular 或其他前端框架来做展示型原型，当前输入已经足够支撑下面这些成果：

1. 可点击的页面导航和路由。
2. 主要流程的线性演示。
3. 核心状态切换，例如空态、执行中、部分完成、失败、中断、恢复。
4. 列表、布局图、详情页、抽屉、弹层、结果页等典型页面构成。
5. 基于假数据或模拟数据的动态演示。
6. 汇报级动画过渡、页面切换和任务进度表现。
7. 一部分交互逻辑，例如表单校验、批量选择、排序、分页、预览汇总、模拟进度。

如果目标是面向评审、汇报或需求对齐，这已经是很高价值的一层。

### 4.2 可以做到比较高保真的内容

当前输入也足够支撑比较高保真的视觉与交互表达，包括：

1. 终端端和平台端各自独立的视觉体系。
2. 拟真的终端列表、教室布局、任务进度、测试结果和告警卡片。
3. 有状态切换的交互页，而不是只有静态线框。
4. 更接近最终产品的按钮、字段、反馈、加载、错误和结果停留样式。
5. 演示用的多页面原型站点或本地原型应用。

只要目标清晰，这一层完全可以做到“看上去已经像一个真实产品”。

## 5. 纯前端原型展示项目不能替代什么

这是最需要提前讲清楚的一点。

### 5.1 不能替代终端本机能力验证

纯前端原型无法真正验证这些东西：

1. 扫描同网段终端。
2. WOL 唤醒与关机。
3. 自动识别当前工作 MAC。
4. 网络参数真实修改。
5. 进入桌面环境、重启返回管理系统。
6. 读取硬件与外设状态。

这些最多只能做交互模拟，不能把它们误当成功能已经验证通过。

### 5.2 不能替代终端与服务器协议验证

纯前端原型可以模拟任务进度和状态变化，但不能真正验证：

1. 实时消息是否稳定。
2. 离线重连补传是否成立。
3. 批量任务取消、中断和断点续传是否成立。
4. 本地完成与平台同步失败时，前后端语义是否一致。

这类问题必须进入后端和终端 Agent 的真实工程链路才能验证。

### 5.3 不能替代部署和容量验证

纯前端原型也不能回答：

1. 一个校区能承受多少终端同时在线。
2. 桌面上传和下发的体量如何落地。
3. 后端长任务执行器怎样调度。
4. 日志、测试结果和资产存储如何组织。

这些都是工程实现问题，不是原型展示问题。

## 6. 如果只做纯前端原型，目前仍然缺哪些环节

虽然当前输入已经足够做原型展示，但若想把原型做得更像真实产品，仍有一些环节值得补充。

### 6.1 视觉系统还没有独立定义

现在的文档已经足够支持页面结构和交互，但还没有独立沉出视觉规范，例如：

1. 终端端与平台端各自的设计语言。
2. 字体、字号、色彩、栅格和间距系统。
3. 列表、卡片、告警标签、进度条、布局图节点等组件风格。
4. 空态、异常态、禁用态和危险操作的视觉规则。

如果缺这层，原型可以完成，但视觉会更像“功能页面集合”而不是“完整产品界面”。

### 6.2 样例数据还不够系统化

高质量展示原型非常依赖样例数据。

当前文档定义了字段和状态，但还没有独立沉出：

1. 典型学校、校区、教室、终端规模样本。
2. 正常、异常、执行中、部分失败、断点续传等任务样本。
3. 平台首页、教室页、终端详情页、测试页、告警页的演示数据集。
4. 终端侧各页的本机信息、桌面列表、网络配置和任务状态样本。

如果补上这层，前端原型会更容易做出“像真实系统”的体验。

### 6.3 动效与演示脚本还没有单独规划

若这个原型要服务于评审和汇报，除了页面本身，还建议补：

1. 默认演示路径。
2. 页面切换节奏。
3. 重点状态切换动画。
4. 演示时用哪组数据讲“单教室部署”“教室维护”“考试前后恢复”“平台下发”等故事。

否则原型即使页面很多，也可能像分散的页面库，而不是一个能讲清产品价值的演示系统。

## 7. 能否给出高保真图

可以，但要先分清“高保真图”具体指哪一层。

### 7.1 如果你说的是高保真页面设计稿

这层可以做到，而且当前输入已经足够支撑：

1. 高保真的页面结构。
2. 接近真实产品的视觉风格。
3. 含状态切换的交互稿。
4. 多页面成套风格稿。
5. 演示级原型站点。

换句话说，我可以把它推进到“非常像最终产品界面”的程度。

### 7.2 如果你说的是像 Figma 那样的设计产物

要分两层看。

我作为 Agent 当前更擅长直接产出的是：

1. Markdown 结构化页面说明。
2. HTML、CSS、Angular 页面原型。
3. 可以本地运行的高保真前端页面。
4. 组件样式、动效和交互逻辑。
5. Mermaid 信息架构图、流程图和结构图。

这意味着我最强的输出不是“直接生成一个真正的 Figma 文件并在 Figma 里形成完整设计系统”，而是：

1. 先把页面和交互做成高保真前端原型。
2. 把视觉、布局、组件和交互关系表达到接近设计稿的程度。
3. 必要时再把这些页面稿转交给设计工具继续整理。

### 7.3 我当前做不到什么程度

在当前工作环境下，我不能直接像设计师在 Figma 里那样：

1. 直接生成真正可编辑的 Figma 设计文件。
2. 直接替你在 Figma 画布上搭组件集和自动布局。
3. 直接输出带完整设计协作元数据的 Figma 项目。

所以，如果你问“能不能直接给到 Figma 那样的可协作设计源文件”，答案是否定的。

### 7.4 我能给到的最接近高保真的成果是什么

我当前最能落地的高保真成果是下面几种：

1. 高保真的前端页面原型，能本地运行、能点击、能切状态。
2. 高保真的静态页面稿，直接以 HTML + CSS 或 Angular 页面输出。
3. 带完整信息架构、页面结构、文案、状态和交互的 Markdown 设计稿。
4. Mermaid 流程图、页面树和关系图，用于补结构说明。

如果目标是帮助研发、产品和业务一起评审，这些成果通常已经比单纯的 Markdown 更有用，也比只给静态图片更接近真实体验。

## 8. 纯前端原型展示项目的推荐产出形态

如果现在先走“纯前端原型展示”路线，更稳妥的产出组合是：

1. 一套结构化页面说明文档，继续承接现有 04、05、06、07、08 的口径。
2. 一套前端高保真原型页面，可本地运行。
3. 一套演示数据和状态样本。
4. 一份演示路径说明，告诉评审如何从首页走到各条主线。

这样既能保留文档推演能力，也能提供足够强的视觉和交互展示。

## 9. 对下一步的实用判断

如果当前目标是先把产品讲清、评审清、汇报清，而不是立即进入完整研发，那么纯前端原型展示完全值得做，而且已经具备开工条件。

更直接的执行判断可以写成：

1. 可以现在开工做纯前端原型展示项目。
2. 可以做到高保真页面和高保真交互演示。
3. 可以做成可本地运行的原型应用，而不只是一组 Markdown 文本。
4. 不能把这类原型误当成终端能力、协议链路和部署能力已经验证完成。
5. 如果后续要继续抬高展示质量，优先补视觉系统、样例数据和演示脚本。
