# 云桌面项目研发可行性与工程落地说明

> 文档日期：2026-03-30
> 文档角色：面向研发工程师的工程落地说明。
> 适用范围：用于判断当前原型与正式基线是否已经足够进入研发阶段，以及进入研发后还需要补哪些工程约束。

## 1. 这份文档解决什么问题

这份文档不替代正式基线，也不替代事实层和页面层。它只回答研发启动前最关键的几类问题：

1. 当前文档输入是否已经足够开始创建代码工程。
2. 当前输入是否已经足够直接完成完整系统实现。
3. 如果技术栈选为终端侧 Electron、平台侧 Chrome、前端 Angular、后端 C#，更稳妥的工程落地方向是什么。
4. 在正式开工前后，哪些工程约束必须补清，否则后续很容易返工。

## 2. 结论先行

当前这套输入可以支持项目正式进入研发阶段，但还不足以在不补工程约束的前提下直接闭环完整实现。

更具体地说：

1. 现在已经足够创建代码工程、拆分模块、搭建前后端基础骨架，并推进到首版可运行系统。
2. 现在还不足以直接锁死全部接口、协议、部署方式和资产模型，因此不适合假设“后面不再补工程规则也能一路做完”。
3. 更稳妥的推进方式不是先把所有页面静态写完，而是先搭工程骨架，再把最影响返工成本的工程约束补齐。

## 3. 为什么说已经足够开工

### 3.1 产品边界已经足够清楚

从正式基线可以拿到稳定的产品边界：

1. 本产品不是重型 VDI，而是以终端本地运行能力为基础、以服务器统一纳管能力为增强的教学云桌面方案。
2. 单教室部署、交付后维护、考试桌面、故障处理、平台纳管这些主线已经拆清。
3. 终端侧负责本地制作、批量部署、本机修正和故障处理；平台侧负责教室创建与终端登记、汇总、跨教室桌面部署、测试、告警和运维入口。

这意味着研发可以直接按终端域、平台域、服务器域和共享领域模型拆工程，而不是还停留在产品范围不清的状态。

### 3.2 核心业务对象已经足够支撑首版领域建模

从业务对象关系图和 PRD 可以稳定提取出首版领域模型的骨架：

1. 终端。
2. 终端业务配置。
3. 终端运行状态。
4. 教室。
5. 服务器端终端登记。
6. 桌面。
7. 基础镜像和快照。
8. 测试结果与告警记录。
9. BIOS 引导策略。

虽然这些对象还没有被细化为数据库表、接口 DTO 或事件协议，但已经足够作为工程里的聚合和模块起点。

### 3.3 页面和主流程已经足够支撑工程分层

当前 06 和 07 已经把页面和流程拆到研发可以直接消费的程度：

1. 终端侧有 TERM-01 到 TERM-07 的稳定主线。
2. 平台侧有 WEB-01 到 WEB-05（含 WEB-02A 终端详情子页）的稳定主线。
3. 终端本地完成与平台统一纳管的边界已经拆开。
4. 批量任务、单机场景、受控任务屏和故障处理的关系已经说清。

这意味着前端工程可以直接开始建立路由、页面域、状态容器和交互骨架，后端工程也可以按终端管理、教室管理、桌面资产、任务执行、测试告警、系统运维等域拆模块。

### 3.4 事实层已经足够支撑首版状态流转设计

当前 05 事实层已经给出了很多研发真正会用到的稳定边界：

1. 当前值、配置值、准备写入值的分层。
2. 终端本地完成与服务器同步失败并行成立的边界。
3. 批量任务里终端范围、保持现状终端和结果停留的口径。
4. 桌面准备与批量桌面选择不是同一层动作。
5. 平台侧与终端侧之间哪些判断能看见，哪些不能反向改写。

这些规则虽然不是接口协议，但已经足够约束前后端状态模型，避免一开始就把状态语义写乱。

## 4. 为什么说还不能直接闭眼做完整实现

当前缺的不是产品原型层，而是工程实现层。风险主要集中在下面几类输入还没有落定。

### 4.1 终端高权限能力层还没有工程化

原型里已经隐含了很多普通前端拿不到的能力：

1. 扫描同网段终端。
2. 自动识别当前工作 MAC。
3. 批量 WOL 开关机。
4. 修改网络参数。
5. 读取硬件与外设状态。
6. 控制重启进入指定桌面环境。
7. 执行替换、重置、桌面上传和桌面下发。

如果只按 Electron + Angular 理解终端侧，这些能力几乎都会卡在权限、系统调用、设备访问或后台长任务执行上。

因此更稳妥的工程判断是：

1. Electron + Angular 更适合承担终端侧 UI 壳和交互编排。
2. 终端本机还需要一层高权限能力服务。
3. 既然后端语言已经选为 C#，终端本机能力层也更适合用 C# Agent 或 Windows Service 承担。

这层如果不尽早设计，后面最容易出现的问题不是页面写不出来，而是很多动作根本没有可靠的执行宿主。

### 4.2 终端与服务器的通信协议还没有锁死

现有文档已经说明了“需要同步什么”，但还没有锁死“怎样同步”。

研发前至少还要补清：

1. 终端状态上报是轮询、长连接还是混合模式。
2. 实时通道是否采用 SignalR，哪些消息必须走实时链路，哪些可以走 REST。
3. 批量任务是命令式下发还是任务式下发。
4. 离线终端重连后的补传与补拉规则。
5. 断点续传、任务取消、任务中断的状态协议。
6. 本地已完成但服务器未同步时，对平台侧返回什么状态码和摘要。

这些问题不解决，前端状态管理、后端任务引擎和终端 Agent 的协作边界都很容易反复返工。

### 4.3 后端基础架构还有关键选型未定

用户已经给了 C# 技术栈，但仍有一些关键工程选型没有落文：

1. ASP.NET Core 是先单体起步，还是一开始拆服务。
2. 数据库使用 SQL Server、PostgreSQL 还是其他方案。
3. 桌面资产、日志、测试结果和上传包存数据库、文件系统还是对象存储。
4. 平台实时通道是否默认以 SignalR 为主。
5. 登录态采用 Cookie、JWT 还是混合模式。
6. 校区隔离、账号可见范围和审计日志如何在数据层落地。

这些不决定“能不能现在开工”，但会直接影响工程目录、基础设施层和接口风格。

### 4.4 终端端与平台端的共享边界还没有定稿

当前可以明确会有不少共享内容：

1. 领域模型和 DTO。
2. 枚举、状态码和任务阶段语义。
3. 一部分表单规则和字段口径。
4. 某些基础 UI 组件和列表展示模式。

但下面这些边界现在还没定：

1. 是否采用 monorepo。
2. Angular 组件库和设计 token 是否终端端、平台端共用。
3. 哪些状态容器和服务可以共享，哪些必须端内独立。
4. Electron 壳层与平台 Web 工程是同仓并排，还是完全分仓。

这类问题不一定需要在第一天全部定死，但至少要在开工初期就形成可执行方案，否则共享代码会很快失控。

### 4.5 非功能指标还不够量化

现有文档对业务边界已经很清楚，但研发还缺少足以指导架构和容量设计的量化约束，例如：

1. 单教室终端规模上限。
2. 单校区教室规模上限。
3. 同时在线终端规模。
4. 同时执行批量任务的上限。
5. 单个桌面资产的典型体量与极限体量。
6. 测试结果、告警和日志的保存周期。
7. 上传、下发、补传和断点续传的并发目标。

这些量化输入会直接影响后端任务模型、存储设计、实时推送压力和终端 Agent 的并发策略。

### 4.6 桌面资产的工程抽象还需要再落一层

产品层已经明确了用户侧统一使用“桌面”，并接受快照式和部分物理分区式两类部署方式并存。但研发还需要继续落文：

1. 工程里是否抽象成统一的桌面包对象。
2. 快照式桌面和物理分区式桌面共用哪些字段，差异字段如何承接。
3. 终端本地资产版本号和服务器资产版本号如何映射。
4. 上传、下发、替换、重置时分别以哪一种资产标识为准。
5. 桌面、基础镜像、快照三者在后端仓储层的关系如何组织。

如果这个抽象不稳，后续最容易出现的是接口名看起来一致，但不同部署方式下语义完全不同。

## 5. 对当前技术栈的工程判断

### 5.1 终端侧

终端侧如果只写成“Electron + Angular”，结论不完整。

更稳妥的落地方向是：

1. Electron 负责桌面壳、窗口管理、宿主能力桥接和本地运行容器。
2. Angular 负责终端 UI、路由、状态管理和页面交互。
3. 本机 C# Agent 或 Service 负责高权限能力、设备访问、网络配置、系统调用和后台长任务执行。

这是当前最接近产品真实能力需求的技术结构。

### 5.2 平台侧

平台侧的“跑在 Chrome 上”本质上是目标运行环境，不是额外工程约束。

工程上更重要的是：

1. 平台侧本质上是 Angular Web 应用。
2. 其核心问题不在浏览器壳，而在页面域拆分、权限范围、实时数据刷新和批量任务预览交互。
3. 如果终端端与平台端都使用 Angular，可以复用部分领域模型、表单规则和组件规范，但仍要避免把终端端特殊交互硬塞到平台端。

### 5.3 后端

后端使用 C# 是合理的，建议优先落在 ASP.NET Core 上，并预留三类能力：

1. REST API，承接增删改查、任务创建、配置写入和结果查询。
2. SignalR 或等价实时通道，承接任务进度、终端状态、告警变化和实时摘要。
3. 后台任务执行器，承接桌面上传、下发、批量动作编排、断点续传和重试。

只把后端理解为常规 Web API 会低估这个项目的长任务和同步要求。

## 6. 研发可以直接开工做到什么程度

在当前输入基础上，研发已经可以比较稳地推进到下面这个深度：

1. 建立多工程或 monorepo 的基础结构。
2. 建立终端端、平台端和后端三套基础工程。
3. 抽出共享领域模型、枚举、任务阶段和基础 DTO。
4. 搭好 Angular 路由、页面壳和基础状态管理。
5. 建立 Mock API 和部分端到端假数据链路。
6. 建立 ASP.NET Core 的模块骨架、任务骨架和接口骨架。
7. 建立 Electron 与本机 Agent 之间的最小通信桥。
8. 先打通少数高价值主线，例如教室列表、终端详情、单教室部署预览、批量任务进度和测试告警摘要。

这是“正式进入研发阶段”可以合理承诺的深度。

## 7. 哪些内容不建议带着模糊状态直接开做

下面这些点如果一直不落文，越往后返工成本越高：

1. 终端 Agent 与后端的通信协议。
2. 批量任务状态机与取消、中断、补传机制。
3. 桌面资产的统一抽象与版本号方案。
4. 平台认证、账号范围与数据隔离实现方式。
5. 大文件上传、下载、断点续传和失败恢复策略。
6. 终端本机能力层与 Electron 壳之间的职责边界。

这些问题不是“可以以后再细化的普通 UI 问题”，而是会决定整体工程走向的架构点。

## 8. 建议的研发推进顺序

更稳妥的顺序不是先试图闭环全部业务，而是按下面方式推进：

1. 先建立工程结构、共享模型和运行骨架。
2. 先把终端端、平台端、后端、终端 Agent 的职责边界定出来。
3. 优先补协议、任务状态机、认证和桌面资产抽象。
4. 先做高价值主线的真链路，再扩外围流程。
5. 在真链路跑通后，再补更多页面细节、导出、运维和审计等外围能力。

这样做的好处是：越早暴露的问题越偏工程根部，越容易控制返工范围。

## 9. 对研发工程师最实用的收口结论

如果把问题收成一句话，就是：

当前输入已经足够开始研发，但研发启动时应默认“先搭骨架、边做边锁工程约束”，而不是默认“原型文档已经细到可以直接一路实现到交付”。

更直接的执行判断可以写成：

1. 可以现在开工。
2. 可以现在搭终端端、平台端、后端和终端 Agent 的首版骨架。
3. 可以现在开始抽共享模型和高价值主线。
4. 不应假设接口、协议、资产模型和部署约束已经完全定稿。
5. 研发开始后的第一批补文重点，应放在协议、任务状态机、认证、存储与桌面资产抽象上。
