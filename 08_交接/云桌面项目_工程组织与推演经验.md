# 云桌面项目_工程组织与推演经验

> 文档日期：2026-03-27
> 文档角色：云桌面项目的工程组织概览和推演经验记录。
> 适用范围：帮助接手者理解本项目的目录分层、推进顺序和已形成的工程习惯；同时记录了可复用到其他项目的推演经验。

## 1. 这份文档解决什么问题

这份文档回答两类问题：

1. 云桌面项目的目录为什么这样组织，各层之间是什么关系，推进的先后顺序是什么。
2. 从这个项目中提炼出的推演经验，哪些可以直接复用到其他原型设计项目。

它不负责定义具体的业务事实，也不替代 AI 交接上下文中的锚点和阅读路径说明。

## 2. 通用项目的推进顺序

### 2.1 先锚定正式基线

如果正式基线还不稳定，就不要急着大面积展开页面。

起点应当是：

1. 先确认需求清单、业务对象关系、PRD、正式约束这些材料是否齐备。
2. 先确定后续所有判断默认服从哪几份文档。
3. 页面、事实和后续总结，都以这些正式锚点为前提继续展开。

这一步要解决的不是“文档够不够多”，而是“后面所有推理到底服从谁”。

### 2.2 再抽稳定事实

正式基线确定后，不要直接把所有判断塞进页面正文。

应先抽一层稳定事实：

1. 把跨页面、跨模块、跨角色会反复用到的规则单独沉出来。
2. 把对象边界、字段口径、状态口径、恢复规则、同步关系单独讲清楚。
3. 这层只写后续页面必须服从的规则，不负责排版某一页。

只要一条判断会同时约束多个页面，它就更适合先进入事实层，而不是散落在若干页面正文里。

### 2.3 再按项目实际拆页面层或模块层

不是所有项目都要拆成终端和平台两端。

按项目真实存在的视角拆层，而不是按模板硬套层级。例如：

1. 单端产品可以按主流程页、设置页、管理页、结果页拆开。
2. 多角色产品可以按操作者角色拆开。
3. 后台系统可以按业务域、对象域、任务域拆开。
4. 跨端项目才需要再按不同端侧分别建总纲和页面正文。

无论具体怎么拆，这一层都只承接页面、流程、按钮、状态、弹层和结果停留，不反向替代事实层。

### 2.4 再做跨层校验与回修

页面一旦展开，就会暴露很多前面看不出来的问题。

这时应按以下顺序处理：

1. 先判断问题属于事实层、页面层，还是未决问题。
2. 如果属于对象边界、字段口径、页面关系，就先回底层规则修正。
3. 再回直接受影响的页面或模块正文同步。
4. 最后再检查是否还有术语残留、按钮漏承接、状态漏闭环。

真正稳定的工程，不是“某一页单独写顺了”，而是上下层能互相对得上。

### 2.5 最后再补接续材料和方法总结

接续材料和方法总结是需要的，但它们不应该抢在事实层和页面层前面生长。

顺序应当是：

1. 正式基线先稳定。
2. 事实层和直接受影响的页面先稳定。
3. 再补当前项目的接手说明、阅读顺序和方法总结。

这样接续材料记录的是已经跑通的方法和已经成立的结构，而不是把尚未确认的推理提前包装成结论。

## 3. 每一层分别写什么

### 3.1 正式基线写什么

适合放在正式基线里的内容通常包括：

1. 需求清单。
2. 业务对象关系。
3. 正式 PRD。
4. 产品级约束与默认语义。

这一层不适合写页面补丁史、阶段回顾或某一轮临时判断。

### 3.2 事实层写什么

适合放在事实层里的内容通常包括：

1. 对象边界。
2. 字段分层和术语口径。
3. 状态口径、恢复规则和同步关系。
4. 会同时约束多个页面或模块的稳定结论。

事实层不展开某一页完整线框，也不承担阶段总结。

### 3.3 页面层或模块层写什么

适合放在页面层或模块层里的内容通常包括：

1. 页面总纲或模块总纲。
2. 页面之间的关系。
3. 当前页字段、按钮、状态、弹层和流程。
4. 当前页的结果停留、失败反馈和返回路径。

这一层要直接写当前页成立的结果，不写“这一轮改了什么”“第一版已经补了什么”这类过程说明。

### 3.4 接续材料写什么

适合放在接续材料里的内容通常包括：

1. 给后续同事或 Agent 的阅读入口。
2. 当前仓库怎样组织。
3. 当前哪些边界已经稳定，哪些地方仍要小心。
4. 已经验证过的工程经验。

它服务的是接续和复用，不应反向变成正式事实前提。

## 4. 通用项目都应守住的写法原则

无论项目是不是跨端，下面这些原则都值得固定下来：

1. 正式正文只写当前成立的结果，不写更新日志式说明。
2. 只要按钮、入口、弹层或查看结果动作出现了，就要继续补清它的承接形式。
3. 当前值、已保存配置值、这次准备写入值必须拆开表达，不能混成一条字段。
4. 如果同一条规则同时影响多个页面，先修事实层，再修页面层。
5. 小改动只按小改动处理，不把一个局部便利动作放大成一整套新机制。

## 5. 云桌面项目是怎样按这套方法推进的

云桌面项目可以作为一个具体例子，但它只是例子，不是模板本身。

### 5.1 起点：先把正式基线固定住

这个项目先把需求清单、业务对象关系和 PRD 固定成正式锚点，然后才继续往下展开。

价值在于：后面每次纠偏时，都能先回到正式锚点，而不是在一堆派生稿之间互相覆盖。

### 5.2 第二步：先沉 05 事实层

在继续画终端页和平台页之前，先把跨页面都会反复用到的边界沉到 05：

1. 字段分层。
2. 页面边界。
3. 批量任务和单机场景的关系。
4. 平台侧、服务器侧、配置、设置等稳定口径。

价值在于：06 和 07 后续修改时，不必每一页重新发明一版底层规则。

### 5.3 第三步：再展开 06 和 07

当 05 足够稳定后，再分别展开终端侧和平台侧页面正文。

这里不是机械地“按端拆”，而是因为这个项目客观上同时存在终端本地主线和平台纳管视角，所以 06、07 分开处理更合理。

价值在于：

1. 终端主线不会被平台汇总视角改写。
2. 平台页也不必反过来重复终端本地完成判断。
3. 跨层冲突出现时，可以先回 05 收平，再同步两边正文。

### 5.4 最后再补接续材料

等正式基线、事实层和页面层已经有了清晰结构，才补当前项目的接手说明和这类方法总览。

价值在于：接续材料记录的是已经跑通的结构和经验，而不是把探索过程直接写进正式正文。

## 6. 云桌面项目给出的经验

这个项目最有价值的经验，不在业务名词本身，而在推进方式：

1. 正式锚点不稳时，不要急着堆页面。
2. 跨页共用规则先沉事实层，页面正文只消费结果。
3. 文档一旦出现更新日志式写法，后面很容易把历史补丁误当成当前结论。
4. 小改动也要尊重真实物理行为，不能只按抽象交互词去想象页面。
5. 页面动作的摆放位置本身就是语义，不能把真实操作按钮藏进说明文字里。

## 7. 云桌面项目里的一个具体例子

这里可以用“修改本机桌面”这个小改动作为一个例子。

真正正确的处理方式，不是把它拔高成一堆新规则，也不是只抓住某个抽象交互词继续扩写，而是回到真实行为本身：

1. 这是一个为了方便用户的小修改。
2. 按钮必须放在合适的位置，用户能直接点到。
3. 点击后终端会真实重启进入当前母机的本机桌面处理；在那里可以新建桌面、复制已有桌面，或进入桌面编辑。
4. 完成这次桌面处理并再次重启后，系统要回到云桌面管理系统，并定位回发起它的原页面与原步骤。

这条经验可以迁移到其他项目里：一旦页面行为和真实系统行为脱节，原型就会越写越假。

## 8. 其他项目如何复用

其他项目即使不是跨端项目，也可以按下面顺序直接复用：

1. 先找出正式基线。
2. 先抽稳定事实。
3. 再按项目真实存在的页面视角、角色视角或模块视角继续展开。
4. 跨层冲突时先修底层规则，再修页面正文。
5. 等结构稳定后，再补接续材料和方法总结。

真正需要替换的，是业务对象、角色结构和页面对象；真正尽量不要替换的，是“先基线、后事实、再页面、最后补接续材料”的推进顺序。

## 9. 推荐阅读顺序

如果是复用方法：

1. 先找当前项目的正式基线。
2. 再看当前项目的事实层。
3. 再看目标页面或目标模块。
4. 最后再看接续材料和方法总结。

如果是接手云桌面项目本身：

1. 先读 04 正式基线。
2. 再读 05 事实层。
3. 再按任务进入 06 或 07。
4. 最后补读当前项目的 AI 交接上下文。
