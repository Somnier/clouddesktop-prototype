# 云桌面管理系统 — 原型设计与深度人机协同实践

[![License: MIT](https://img.shields.io/badge/License-MIT-blue.svg)](LICENSE)

本项目包含两条并行主线：

1. **云桌面管理系统的完整产品原型设计**——从原始需求到可运行的交互原型，覆盖终端侧（母机管理软件）和平台侧（Web 管理平台），管理多校区、多教室、多终端的桌面生命周期。原型以导演台、母机终端、受控终端、管理平台四端同屏运行，共享同一份状态并通过 SSE 实时同步。
2. **深度人机协同的工程实践**——以该产品设计为场景，探索并记录人与 AI Agent 在多轮、多文件、多阶段项目中的协作方法。

云桌面产品的设计方向是通用终端管理，初始需求仅研究了高校教学场景。

---

## 项目结构

项目按"正式基线 → 事实层 → 页面层 → 原型 → 交接"的顺序递进，每一层的产出沉淀在对应目录：

```
├── 01_原始资料/           # 客户侧原始需求清单
├── 02_方法指南/           # 原型设计与人机协同的经验总结 + AI Agent 工作模式分析
├── 03_探索稿/             # 业务建模迭代过程（v1-v11 + PRD 探索版本）
├── 04_正式基线/           # PRD、业务对象关系图、需求清单（冻结基线）
├── 05_云桌面事实/         # 跨页面、跨端的业务事实与稳定结论
├── 06_终端侧页面设计/     # 终端管理软件 TERM-01 ~ TERM-07
├── 07_平台侧页面设计/     # Web 管理平台 WEB-01 ~ WEB-05 + WEB-02A
├── 08_交接/               # AI 交接上下文、可行性分析、项目概览、仓库事实记忆镜像
├── 09_云桌面原型/         # 完整交互原型（Node.js + Express + SSE + Electron）
├── 10_云桌面原型简化版v1/ # 简化版原型 v1
└── 11_云桌面原型简化版v2/ # 简化版原型 v2
```

01 是客户侧的原始需求清单，按特性分级记录了功能描述和优先级。03 记录了从原始资料出发的 11 轮业务建模迭代，收敛为 04 正式基线。05 的事实文档在页面设计过程中逐步沉淀，与 06/07 交替推进、互相校验。08 面向项目接续，包含给后续 AI Agent 或人类的上下文说明和可行性分析。

## 产品设计概要

### 系统定位

不是重型 VDI，而是以终端本地运行能力为基础、以服务器统一纳管能力为增强的云桌面方案。核心设计原则：

- 支持脱机交付：网络未就绪时，单教室仍可独立完成部署
- 联机后增强：终端状态、桌面资产和异常信息逐步纳入统一管理
- 多教室复用：重复单教室流程为基础路径，服务器辅助和离线打包并列提效
- 交付后自助：机房管理员在远程指导下即可完成常见运维

### 终端侧页面（06 目录）

终端管理软件运行在教室母机上，共 7 个页面：

| 编号 | 页面 | 说明 |
|---|---|---|
| TERM-01 | 管理系统首页 | 教室状态总览、终端列表、网络与服务器配置 |
| TERM-02 | 母机接管页 | 接管教室、释放教室、换母机识别 |
| TERM-03 | 母机工作页 | 桌面制作、批量日常操作的中枢 |
| TERM-04 | 单教室部署 | 3 步向导：选来源 → 选桌面 → 选目标终端 → 执行 |
| TERM-05 | 教室维护 | 加装软件、IP 修改、教室终端清单导出 |
| TERM-06 | 故障处理 | 一键替换、一键重置、手动修正 |
| TERM-07 | 考试桌面 | 考试启动设置与考后恢复 |

### 平台侧页面（07 目录）

Web 管理平台运行在服务器上，共 5 个主页面 + 1 个详情子页：

| 编号 | 页面 | 说明 |
|---|---|---|
| WEB-01 | 平台总览 | 校区级指标、服务器状态、任务进度、异常与风险 |
| WEB-02 | 教室管理 | 教室列表、终端清单导入、批量工具栏（分 4 组：单教室操作、单教室测试、多教室操作、多教室测试） |
| WEB-02A | 终端详情 | 单终端配置、硬件、网络、桌面、告警全量信息 |
| WEB-03 | 桌面资产 | 校区级桌面管理，支持桌面包导入导出和无引用桌面清理 |
| WEB-04 | 告警中心 | 按教室/类型筛选与排序、跳转教室和终端 |
| WEB-05 | 系统运维 | 授权管理、服务器地址变更、校区与账号管理 |

### 事实层（05 目录）

设计过程中沉淀的业务事实，供所有页面引用而不是各自重复推导：

- 母机接管与教室绑定机制
- 批量部署范围与终端状态同步机制
- 任务执行中断与受控终端任务屏机制
- 终端与服务器按需同步
- 平台侧页面范围与任务边界
- 跨端页面表达与字段口径

## 交互原型

导演台、母机终端、受控终端、管理平台四端共享同一份 JSON 状态，通过 SSE 实时同步：

| 端 | 地址 | 说明 |
|---|---|---|
| 导演台 | `http://localhost:3920/director` | 测试控制台：切换场景、触发动作、模拟故障 |
| 母机终端 | `http://localhost:3920/terminal/mother` | 终端管理软件的完整交互 |
| 受控终端 | `http://localhost:3920/terminal/controlled` | 受控终端任务屏 |
| 管理平台 | `http://localhost:3920/platform` | Web 管理平台 |

### 快速启动

```bash
cd 09_云桌面原型
npm install
npm run serve          # Web 服务器（浏览器访问）
# 或
npm start              # Web + Electron 桌面窗口
```

种子数据（`data/seed.json`）包含模拟的教室和终端数据，导演台可随时重置。

## 经验总结（02 目录）

项目推进过程中积累了三份经验性文档：

- **AI Agent 工作模式说明**：对 VS Code + Copilot Agent 环境的工程化分析，记录了从纯对话到 Agent 编排的能力路径、工具体系和记忆机制。
- **原型设计工程实践指南**：从原始资料到可运行原型的五阶段工程经验（原始资料整理 → 事实层固化 → 页面结构设计 → 可交互原型实现 → 评审与修正）。
- **深度人机协同工程实践指南**：与 AI Agent 进行多轮、多文件、多阶段深度协作的工程经验（事实分层、锚点机制、偏移控制、边界定义）。
