# 云桌面业务事实索引

> 文档角色：业务事实层的索引页。
> 作用：登记本层全部业务事实文档，并约定"事实如何写、写什么、何时写"。本层把已确认、且会跨页面/跨端约束设计的业务机制集中沉淀，作为页面设计与原型实现共同的认知底座。

## 1. 本层放什么

业务事实承接的是已确认、并会直接约束设计与实现的**业务机制与口径**——业务在做什么、有哪些核心对象、对象之间的关系与边界。

满足以下任一条件的机制，应单独落成一份业务事实文档：

- **跨页面约束**：同时影响两个及以上页面，否则每页都要重复推导。
- **跨端约束**：涉及终端侧与平台侧之间的数据口径、展示差异或同步边界。
- **术语与字段统一**：定义多份文档共用的术语、字段名称或状态枚举。

仅在单一页面内部生效的交互细则，写在对应页面文档即可，不上提到本层。**界面呈现与交互方式**层面的事实（颜色语义、排版基准、表单 / 弹框规范、交互流程、范围取舍）不在本层，统一沉淀在 [设计事实](../04_云桌面设计事实/云桌面设计事实索引.md)。

## 2. 事实文档怎么写、何时写

1. **何时写**：在澄清需求、推进设计、修改实现的同时同步沉淀或校对事实——发现既有事实不足或与认知不符时，立即回到本层补全 / 订正，而不是把结论只留在记忆或代码里。
2. **自上而下推演式修改，不打补丁**：事实文档表达的是当前成立的完整结论，不体现"某次改了什么、为什么改"。修订时重写相关章节使其自洽，严禁以"补充：本次把 X 改为 Y""注意：之前的 Z 已废弃"等补丁式、强调式语句堆叠修改痕迹与修改原因。读者只应读到当前事实。
3. **重写即正名**：当一份文档被实质性重写、主题或范围已变时，文件名随内容同步更名，使文件名始终准确描述其当前内容。
4. **概念锚定统一口径**：指代同一对象时一律沿用 [云桌面核心概念与统一口径](./云桌面核心概念与统一口径.md) 的术语与口径，不另造词。

## 4. 与其它层的衔接

1. **正式基线** 是产品经理视角的需求描述；业务事实把它转化为产品认知与设计基石。
2. **设计事实** 承接界面呈现与交互方式；业务事实与设计事实并列，共同作为页面设计与原型实现的认知底座。
3. **终端侧 / 平台侧页面设计** 引用业务事实与设计事实取口径；**原型** 是可运行实现，其早期可行性验证随原型一并归档。
