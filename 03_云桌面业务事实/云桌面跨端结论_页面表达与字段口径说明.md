# 云桌面跨端结论_页面表达与字段口径说明

> 文档角色：跨端稳定结论之一。
> 作用：把终端侧和平台侧都需要沿用的页面表达原则、字段口径和流程共识放在一处，避免两边各自重写一套。

## 1. 这份文档解决什么问题

终端侧和平台侧虽然承接的页面不同，但它们经常会面对同一批对象、同一批字段和同一批任务结果。如果这些地方没有统一口径，后面很容易出现两种问题：一边把某个字段当当前状态，另一边把它当长期配置；或者一边把某个动作看成整批任务，另一边又把它写成单机动作。

这份文档不负责讲页面怎么排版，也不负责展开某一页正文。它只负责把已经确认的共识写清楚，后续谁需要用到，就直接沿用。

## 2. 数据至少要分成几层

当前项目里的页面数据，至少要区分下面几层：

1. 当前实时状态，例如当前在线、当前机器名、当前 IP、当前工作 MAC、当前连通性。
2. 终端业务配置，例如教室、座位号、终端用途、机器名配置、网络配置、服务器地址、默认启动项和还原方式。
3. 当前任务数据，例如这轮勾选了哪些终端、当前顺序是什么、这次准备写入什么。
4. 当前任务运行状态，例如当前步骤、当前进度、失败原因、同步状态。
5. 服务器侧记录或平台侧汇总数据，例如终端登记信息、桌面资产状态、平台汇总状态。

只要页面里同时出现了这几层数据，就必须把它们拆开表达，不能用一条字段代替多层含义。

补充四条稳定口径：

1. “平台侧”用于 Web 页面、平台账号操作和平台汇总视角。
2. “服务器侧”用于服务器本体动作、服务器端登记信息和服务器侧记录的数据。
3. “配置”用于持久业务值或服务器端/终端登记信息中应长期保存的字段。
4. “设置”用于用户当前正在选择、确认或准备写入的动作条件与任务参数。

如果同一页同时出现“当前值、配置值和这次设置值”，应明确写成三层，不要把“设置”直接当成已经落库的“配置”。

## 3. 当前值、配置值和这次要写入的值不能混成一条

网络相关字段最容易被写乱，但这个原则并不只适用于 IP。

后续页面凡是同时涉及“现在是什么”“长期应该是什么”“这次准备改成什么”，都要明确区分：

1. 当前生效值。
2. 业务配置值。
3. 这次任务准备写入的值。

当前 IP、网络配置里的 IP、这次准备批量写入的 IP，分别属于这三层；机器名、默认启动项、还原方式、终端用途在很多页面里也是同样的关系。页面可以把它们放在一起对照，但不能把它们混写成一个模糊字段。

这组字段也不是批量页专属。机器名、座位号、终端用途、网络配置、服务器地址、桌面集合、默认启动项和还原方式，在批量页和单机页里表达的是同一组业务字段；批量页负责整批写入，TERM-01 的本机页负责单台处理。

## 4. 为什么要同时定义母机页、受控任务屏和 TERM-06 的边界

这里的“一起定义”不表示它们属于同一条正在执行的流程，也不表示用户可以在这几个入口之间随意来回跳转。

真正需要放在一处说清的，是下面这组边界：

1. 母机页负责整批任务视角，关心的是整间教室、整批终端和整轮任务结果。
2. 受控终端任务屏负责本机视角，关心的是本机当前被谁控制、有没有被纳入、执行到哪一步。
3. TERM-06 是单终端故障处理入口，负责替换、重置和网络修正这类单机问题，但它不是批量任务执行中的并行分支。

因此这三者需要一起定义的，不是“它们如何同时协作”，而是“它们什么时候互斥、什么时候切换、什么时候绝对不能切换”：

1. 在批量任务执行中，母机页和受控任务屏属于同一轮任务的两个视角。
2. 在受控执行中，受控终端不允许自行跳去 TERM-06；否则控制和被控制关系就被打断了。
3. TERM-06 只在终端未受控、当前任务已中断、当前任务已取消，或该终端已经明确脱离整批任务后才成立。
4. 如果不把这几条边界放在一起说明，页面很容易被写成“受控过程中还能自行一键重置/一键替换/改本机设置”，这会直接破坏任务流程本身。

所以这里真正要统一的是边界，不是把三者写成同一功能。

## 5. 术语怎么用才不会半改半留

当前项目里，机制术语和 UI 文案可以不完全一样，但不能半句一个口径。

更合适的做法是：

1. 在项目事实和页面正文里，继续稳定使用“母机”“受控终端”“教师终端”“学生终端”等已经有业务含义的术语。
2. 在按钮名、入口名和部分状态名里，可以使用更直接面向用户的表达，例如“进入教室管理模式”而不是“进入母机工作模式”。
3. 如果某一页标题、正文说明、按钮文案分别用了三套不同叫法，用户和研发都会误解。

所以后续文档里不再大面积更换已经稳定的业务术语。需要解释时就解释清楚关系，不靠换词制造新口径。

补充两条当前可见性口径：

1. MAC 地址是系统内部的关键标识，必须保留；但用户阅读 MAC 地址困难，因此 UI 主路径应尽量弱化展示，转而用座位号、机器名等业务概念替代；详情页和诊断场景仍应保留完整 MAC 信息。
2. 桌面同步状态统一为"未同步 / 同步中 / 已同步"三态；"未同步"需要显著提示，避免被误认为已完成服务器侧可复用。

补充一条座位号与机器名格式口径：

1. 座位号格式为字母+数字连写，不含横杠。例如：A01、B12、T01。
2. 机器名由教室编号前缀和座位号以横杠连接。例如：D301-A01、A301-T01。
3. 座位号和机器名是两个独立字段，座位号不继承机器名的横杠格式。

## 6. 04、05、07 的桌面下发集合都可以多选，但复制和编辑仍然是单桌面动作

这条边界必须固定下来，否则部署、维护和考试三类流程都会越写越乱。

更准确的结论是：

1. 桌面编辑是单桌面动作。
2. 复制桌面是单桌面动作。
3. 新建桌面虽然不依赖来源桌面，但创建完成后，当前继续处理的仍然是一个具体桌面。
4. TERM-04、TERM-05、TERM-07 这类桌面相关教室级任务里，待下发桌面集合都可以多选。
5. 默认启动桌面或考试启动桌面，应从当前勾选的桌面集合里再选 1 个。
6. 批量部署、批量维护、考试前设置和考后恢复，作用对象都是一批终端。

因此，后续凡是页面里同时出现“选桌面”和“选终端”两类动作时，都要把它们分清：桌面下发集合可以多选；复制和进入编辑只在当前恰好单选 1 个桌面时可用；终端侧才是批量勾选、排序、预览和分发。不要再额外发明“当前操作桌面”这类容易制造歧义的中间概念。

补充说明：

1. TERM-04 的 [修改本机桌面] 放在当前母机准备区；TERM-05、TERM-07 的 [修改本机桌面] 放在各自的桌面选择步骤里，作为单独按钮出现。
2. 点击 [修改本机桌面] 后，终端重启进入当前母机的本机桌面处理；在这里可以新建桌面、复制已有桌面，或进入某个桌面继续编辑。完成这次桌面处理并再次重启后，系统回到云桌面管理系统，并定位回发起它的原页面与原步骤。
3. 这个按钮用于进入当前母机的本机桌面处理，不表示把新建、复制、编辑这些单桌面动作并进批量桌面选择层。

## 7. 这轮勾选终端和整间教室受管集合不是一回事

单教室任务里，整间教室受管集合始终是母机当前看到的整批终端；这轮勾选终端只是其中这一次真正要处理的那一批。教师终端、待单独处理终端和这次故意跳过的终端，仍然可以留在受管集合里，只是不参与这一轮动作。

因此页面里要同时能表达三件事：

1. 当前这间教室总共有多少受管终端。
2. 这轮勾选了哪些终端。
3. 哪些终端这轮保持现状。

只要这一层没分清，母机页、受控任务屏和导出提示就会全部被带偏。

补充两条默认显示原则：

1. “保持现状”不应只给一个抽象标签；默认还要告诉用户这一项本次不修改，以及它实际沿用了什么当前内容。
2. 执行中的教室级任务和任务结果页都应停留在各自任务页，不通过 TERM-03 再承接所谓收口态。

## 8. 批量任务都应该有逐台确认层

只要任务会批量改写终端配置、桌面或启动设置，就不适合在“设置完规则”后直接执行。

更合适的做法是：

1. 先勾选终端并设置规则。
2. 把规则展开成逐台终端的写入结果。
3. 再让用户逐台核对或微调。
4. 确认后才开始真正分发。

这条共识同样适用于部署、维护和考试流程。区别只在于每类流程展开的字段不同，不在于要不要这层逐台确认。

## 9. 本地完成和服务器同步是两条并行线

终端侧本地任务是否完成，应先看本地结果；服务器是否已经接收到状态、配置或桌面资产，是另一条同步线。

因此后续页面必须守住两条判断：

1. 本地部署、维护、考试设置和故障处理可以先按本地结果成立。
2. 服务器上传、统一纳管和跨教室复用，会影响平台侧查看和后续能力，但不反向推翻终端本地已经完成的结果。

只要这两条线被混到一起，后面不是把服务器写成终端前提，就是把本地结果误写成已经全局同步成功。

## 10. 凡是需要人工勾选的列表，都应支持排序和分页

只要页面里存在人工勾选终端、桌面、教室或其他对象的列表，就不应只给一个静态长列表。

这类列表应当：

1. 这类列表都应支持排序。
2. 这类列表都应支持分页。
3. 可排序字段、默认排序字段和分页策略，不在跨页规则层统一写死。
4. 它们应按页面对象类型、当前任务侧重点和数据量分别确定。

这样后续页面既能统一保证可操作性，又不会把所有页面都收成同一个机械模板。
